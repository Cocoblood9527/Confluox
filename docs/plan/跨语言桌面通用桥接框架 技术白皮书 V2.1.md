# 跨语言桌面通用桥接框架 技术白皮书 V2.1

### Universal Cross-Language Desktop Bridge

（面向资深工程师的技术白皮书 · V2.1 修订版）

---

## 0. V2.1 修订说明

本版本基于 `docs/plan/跨语言桌面通用桥接框架 技术白皮书 V2.md` 与其评审报告反向修订，重点修正以下问题：

- 收缩过强承诺，明确真实交付边界；
- 重写启动握手方案，**不再依赖 Uvicorn 人类可读日志解析端口**；
- 补齐流式协议的鉴权模型，显式覆盖 **HTTP / WebSocket / SSE**；
- 重构插件模型，不再默认所有插件都必须落到“同进程 + OpenAPI 聚合”；
- 明确前端选型约束：**默认推荐 Vite + React SPA**；若使用 Next.js，仅支持 **静态导出子集**；
- 增加 **MVP 分阶段落地策略**，避免 V1 阶段范围失控。

V2.1 的目标不是把愿景缩小，而是把白皮书从“雄心勃勃的方向文档”修正为“可以指导实施的工程蓝图”。

## 1. 引言（Introduction）

过去十年，桌面软件开发长期处在一种结构性割裂中：

- **Web 前端生态**擅长构建现代 UI，但默认运行在受限环境中，难以无缝复用本地 Python 能力；
- **Python 生态**拥有最丰富的 AI、自动化、数据处理能力，但桌面交付、依赖管理、跨平台分发成本极高；
- 大量优秀的 Python 项目天然更像“脚本”“CLI”“本地服务”，而不是“可分发的桌面产品”。

这带来一个持续存在的工程问题：

> 如何把已有 Python 能力，以较低改造成本接入现代桌面应用，并尽可能保持统一的宿主、统一的生命周期、统一的安全边界与统一的调用体验？

V2.1 对这个问题的回答，不再是一个绝对化承诺，而是一个分层的基础设施方案：

- 对**轻量 API 能力**，提供近似“即插即用”的桌面桥接；
- 对**可被适配的第三方项目**，提供受约束的适配器模型；
- 对**复杂独立服务**，提供代理/清单/桥接式接入，而不是强行同进程整合；
- 对**本地 MCP Server**，提供专门的协议桥接和生命周期托管。

## 2. 产品定位与承诺边界（Positioning & Delivery Boundary）

### 2.1 核心定位

本框架的定位是：

> 一个用于托管本地能力的跨平台桌面宿主与桥接框架，而不是“任意 Python 项目零成本桌面化”的魔法系统。

它优先解决以下问题：

- 本地桌面宿主与本地 Python 能力之间的结构化桥接；
- 多类本地能力的统一发现、启动、调用、退出与诊断；
- 打包后环境下的数据目录、静态资源、子进程、鉴权、前端调用一致性；
- 为插件式扩展建立一套可以约束复杂度的接入契约。

### 2.2 分层承诺模型

V2.1 不再使用“任何 Python 开源生态，一天内桌面化”这类绝对承诺，而采用分层承诺：

#### A 类：自研 API 插件（最低接入成本）

- 形态：`APIRouter` / 小型 FastAPI 路由；
- 特征：接口明确、依赖可控、静态资源少；
- 承诺：可以接近“低成本快速接入”。

#### B 类：可适配第三方项目（中等接入成本）

- 形态：可通过 `adapter.py` 明确注入配置、数据路径与资源路径的第三方项目；
- 特征：有明确边界、依赖基本兼容；
- 承诺：支持接入，但**不承诺零改造**。

#### C 类：复杂服务型插件（高接入成本）

- 形态：更像独立服务或独立系统，而非可直接 import 的库；
- 特征：自带前端、强状态、强耦合依赖、复杂流式协议；
- 承诺：优先采用独立进程/代理/桥接方式，不强行同进程整合。

#### D 类：本地 MCP Server

- 形态：stdio + JSON-RPC 的本地工具/服务；
- 特征：天然独立进程，更适合桥接而非源码级拼接；
- 承诺：提供协议适配、工具发现、生命周期管理。

### 2.3 In-Scope

V2.1 的正式范围包括：

- 基于 Tauri 的跨平台桌面宿主；
- 基于 Python 的本地网关进程；
- 三类插件模型：**API 插件 / 服务型插件 / MCP 插件**；
- 统一的数据目录注入与静态资源定位机制；
- 统一的进程树管理与优雅停机协议；
- 统一的本地鉴权与前端调用模型；
- 分阶段构建与打包策略。

### 2.4 Out-of-Scope

当前版本明确不解决：

- 云端多租户、RBAC、组织级权限模型；
- 同进程内多个互相冲突的大型 Python 依赖树共存；
- “不写适配器即可接入任意第三方大项目”；
- “所有第三方项目都必须产出 OpenAPI 并进入统一 SDK”；
- 远程分布式编排、集群治理、云端控制平面。

## 3. 架构总览（System Overview）

### 3.1 四层结构

整个系统由四个边界清晰的层组成：

1. **桌面宿主层（Tauri / Rust）**
   - 启动与托管本地网关进程；
   - 注入数据目录、鉴权信息、就绪信号路径；
   - 管理关闭事件与生命周期收束。

2. **前端展示层（Vite + React SPA 为默认推荐）**
   - 只负责展示与交互；
   - 不承载底层业务逻辑；
   - 通过统一 SDK、Manifest 或流式客户端与网关通信。

3. **本地网关层（Python / FastAPI）**
   - 提供系统接口；
   - 挂载 API 插件；
   - 代理服务型插件；
   - 适配 MCP Server；
   - 托管统一鉴权、进程管理、配置注入。

4. **打包与分发层（Build System）**
   - 负责 Python 网关的冻结/打包；
   - 负责静态资源与插件元数据纳入产物；
   - 与 Tauri 安装包集成。

### 3.2 三类插件是平级模型

V2.1 的关键修正是：**不再把所有插件都当成同一种东西。**

#### 插件 A：API 插件

- 接入方式：`setup(app)` / `APIRouter`
- 部署方式：同进程
- 文档能力：优先支持 OpenAPI
- 适合：自研工具、轻量内部 API、可控逻辑

#### 插件 B：服务型插件

- 接入方式：适配器 + 代理 + Manifest
- 部署方式：可同进程，也可独立进程
- 文档能力：**不强求完整 OpenAPI**
- 适合：大型第三方项目、自带前端或特殊协议的服务

#### 插件 C：MCP 插件

- 接入方式：MCP 适配层
- 部署方式：独立进程优先
- 文档能力：基于工具清单与参数 Schema
- 适合：本地工具链、自动化工具、数据库与文件系统操作

### 3.3 OpenAPI 的角色重新定位

V2.1 中，OpenAPI 的定位是：

- **API 插件的一等公民接口描述方式**；
- **前端强类型 SDK 的主要来源**；
- **不是所有插件的统一强制基线**。

这意味着：

- API 插件应尽量进入统一 OpenAPI 聚合；
- 服务型插件可以只暴露能力清单、代理入口或局部 API；
- MCP 插件主要依赖工具 Manifest，而不是强行转换为完整业务 OpenAPI。

## 4. 前端技术栈与约束（Frontend Strategy）

### 4.1 默认推荐：Vite + React SPA

V2.1 将 **Vite + React SPA** 设为默认推荐前端方案，原因如下：

- 与 Tauri 的静态资源分发模型天然一致；
- 更少引入“开发期可用、打包期失效”的误导性能力；
- 对桌面壳场景更直接，心智负担更低；
- 适合作为宿主 UI、插件面板、系统设置、调试视图等纯前端界面。

### 4.2 Next.js 的定位调整

若团队已有 Next.js 资产，可继续使用 Next.js，但必须满足以下前提：

- 仅使用 **静态导出子集**；
- 产物必须是可被 Tauri 直接加载的静态文件；
- 不依赖 Node 服务端运行时；
- 不依赖请求时服务端上下文。

### 4.3 在 Tauri 中使用 Next.js 的硬约束

若选择 Next.js，以下能力默认视为**禁用项**：

- SSR 页面能力；
- Route Handlers 作为运行时后端；
- Middleware / Rewrites / 依赖服务端环境的重写逻辑；
- 依赖服务端上下文的 Server Actions；
- 把 Next.js 当成“桌面内嵌 Node 平台”使用。

可接受的使用方式是：

- 仅借用其组件组织、路由组织、构建工具链；
- 构建产物以静态导出为准；
- 所有后端能力统一通过本地网关提供。

### 4.4 前端调用模型

前端调用分三类：

1. **普通 HTTP JSON**
   - 使用统一 SDK 或 fetch 封装；
   - 适用于系统接口、普通 API 插件。

2. **WebSocket**
   - 适用于终端、对话流、推送型状态同步；
   - 鉴权由 query/subprotocol 处理。

3. **SSE / Fetch Streaming**
   - V2.1 默认推荐 **`fetch + ReadableStream`**；
   - 仅当明确采用签名 URL 或 cookie 方案时，才使用原生 `EventSource`。

## 5. 本地网关与插件层（Gateway & Plugin Runtime）

### 5.1 网关职责

本地网关负责：

- 系统探活、版本信息、诊断信息；
- 插件发现、挂载、路由分发；
- 流式协议入口；
- 数据目录、资源路径、鉴权信息的统一注入；
- 子进程生命周期与优雅停机。

### 5.2 插件契约

V2.1 推荐使用 Manifest + 入口函数的混合契约：

```plaintext
plugins/
  plugin_xxx/
    manifest.json
    entry.py
```

其中：

- `manifest.json` 声明插件类型、入口、显示名称、能力类型、资源目录；
- `entry.py` 暴露 `setup(context)` 或等价接口；
- `context` 由框架提供，包含 `app`、`data_dir`、`auth`、`process_manager`、`resource_resolver` 等能力。

### 5.3 API 插件契约

API 插件必须：

- 暴露稳定的 HTTP API；
- 尽量产出 OpenAPI；
- 不依赖全局 `cwd`；
- 显式使用框架注入的 `data_dir` 与资源解析器。

### 5.4 服务型插件契约

服务型插件不强求进入统一 OpenAPI 聚合，而是必须明确下列边界：

- 启动方式：同进程或独立进程；
- 对外入口：HTTP 代理 / WebSocket 代理 / 内嵌页面 / 局部 OpenAPI；
- 鉴权方式；
- 资源目录与数据目录如何注入；
- 生命周期是否受宿主关闭控制。

V2.1 明确要求：**服务型插件默认优先用适配器隔离复杂度，而不是把第三方原应用直接裸挂到主系统。**

### 5.5 MCP 插件契约

MCP 插件由网关统一适配，至少包括：

- 工具列表发现；
- 参数 Schema 暴露；
- 调用入口；
- 生命周期托管；
- 错误与日志转发。

## 6. 启动握手与生命周期管理（Startup & Lifecycle）

### 6.1 V2.1 的核心原则

V2.1 明确放弃以下脆弱方案：

- Rust 预占端口后再释放给 Python；
- 通过解析 Uvicorn 人类日志获取真实端口；
- 依赖 stdin 管道断裂判断宿主死亡；
- 依赖 `sys.exit()` 或信号模拟完成优雅停机。

### 6.2 正式启动模型

正式启动流程如下：

1. Tauri 宿主生成：
   - `data_dir`
   - `auth_token`（可选但强烈建议）
   - `ready_file` 或等价结构化握手路径
   - `host_pid`
2. 宿主启动 Python 网关 sidecar，并传入上述参数。
3. Python 网关：
   - 解析参数；
   - 初始化配置、插件、鉴权中间件、进程管理器；
   - **在 Python 进程内部完成端口绑定**；
   - 将真实端口、PID、版本、状态写入 `ready_file`。
4. Rust 轮询 `ready_file` 成功后，再把端口与鉴权信息暴露给前端。
5. 前端初始化 SDK / Manifest / 流式客户端。

### 6.3 端口发现策略

V2.1 的正式要求是：

> **端口发现必须依赖结构化握手，不得依赖 Uvicorn 启动日志文本。**

推荐两种实现方式，任选其一：

#### 方案 A：Python 内部预绑定 socket + ready file

- Python 进程在本进程内创建监听 socket，绑定 `127.0.0.1:0`；
- 从 socket 得到真实端口；
- 将 socket 交给程序化创建的 Uvicorn Server；
- 同时把端口写入 `ready_file`。

优点：

- 无 TOCTOU；
- 不依赖 stdout；
- 不依赖日志级别与日志格式。

#### 方案 B：程序化启动后由应用状态回写就绪文件

- 使用 `uvicorn.Server` 编程式启动；
- 在 server 已成功绑定后由框架层回写结构化就绪文件。

此方案同样可行，但实现复杂度略高。

### 6.4 宿主猝死检测

V2.1 正式采用：

- **`--host-pid` + 轮询宿主存活**

原因：

- 不依赖 stdin 管道；
- 跨平台可实现；
- 可与程序化 `uvicorn.Server` 的 `should_exit` 机制配合。

### 6.5 优雅停机协议

正式流程如下：

1. 用户关闭窗口；
2. Rust 拦截关闭事件；
3. Rust 调用 `POST /api/system/shutdown`；
4. Python 网关设置 `server.should_exit = True`；
5. FastAPI / lifespan 执行 shutdown 收尾；
6. `ProcessManager.terminate_all()` 清理子进程；
7. Rust 等待短时间后退出应用。

该协议是强制契约，不得省略。

## 7. 鉴权与本地安全边界（Security Model）

### 7.1 基本原则

本框架默认运行于单机本地场景，但“本地”不等于“安全”。因此必须同时处理：

- 本地恶意网页跨域调用；
- 端口被探测后的未授权访问；
- 流式连接绕过普通 Header 鉴权；
- 插件与子进程越权访问。

### 7.2 网络边界

网关默认只监听：

- `127.0.0.1`

并要求：

- CORS 不得使用 `["*"]`；
- 允许的 Origin 必须由当前 Tauri WebView 的**真实生产 Origin** 决定；
- 具体 Origin 值不写死在规范文档中，而由实现层按平台配置。

### 7.3 鉴权统一模型

V2.1 对不同协议采用明确分流：

#### HTTP

- 使用 `Authorization: Bearer <Token>`。

#### WebSocket

- 使用 query token 或 subprotocol token；
- 不要求浏览器自定义 Header。

#### SSE

V2.1 正式要求必须二选一：

##### 方案 A：默认推荐

- 使用 `fetch + ReadableStream` 实现流式拉取；
- 继续沿用 `Authorization` Header。

##### 方案 B：兼容方案

- 若必须使用原生 `EventSource`；
- 则必须使用短时效 query token、一次性签名 URL 或 cookie 方案；
- 不得让 SSE 端点在无鉴权状态下对外可用。

### 7.4 插件信任模型

V2.1 的插件信任模型仍然是：

- 用户或团队对挂载的插件代码承担信任责任；
- 框架主要提供运行时边界、生命周期治理与最小必要防护；
- 高风险插件如需更强隔离，应使用独立进程、容器、受限运行环境等外部机制。

## 8. 数据目录与资源路径（Data & Resources）

### 8.1 数据目录原则

宿主通过参数注入 `data_dir`，网关只负责：

- 存储到统一配置；
- 下发给插件上下文；
- 不全局修改工作目录。

### 8.2 明确禁止全局 `os.chdir(data_dir)`

V2.1 明确规定：

> **禁止在主进程级别执行全局 `os.chdir(data_dir)`。**

原因：

- 会破坏打包环境中静态资源读取路径；
- 会把第三方对相对路径的静态资源读取误导到数据目录；
- 会造成开发环境与打包环境行为不一致。

### 8.3 静态资源解析器

框架必须提供统一 API，例如：

- `get_resource_path(relative_path: str) -> str`

其职责是兼容：

- 源码环境；
- PyInstaller 打包环境；
- Nuitka 打包环境。

插件不得直接硬写：

- `sys._MEIPASS`
- 假定某种固定 `cwd`

### 8.4 第三方项目的路径注入

对服务型插件或大型第三方项目，路径策略统一如下：

- 数据落盘路径通过适配器显式注入；
- 静态资源读取通过框架资源解析器或适配器提供的资源基址完成；
- 若第三方有稳定配置对象，可用局部 Monkey Patch；
- 若第三方未暴露稳定配置，则必须通过适配器隔离，而不是诉诸全局 Patch。

## 9. OpenAPI、Manifest 与前端集成（Contract Layer）

### 9.1 API 插件：OpenAPI 优先

API 插件应尽量进入统一 OpenAPI 聚合，以便：

- 生成强类型 SDK；
- 提供一致的前端调用体验；
- 支持自动文档与调试。

### 9.2 服务型插件：Manifest 优先

服务型插件不强求完整 OpenAPI，而使用如下契约：

- 插件名称；
- 启动方式；
- 能力类型；
- 可选入口地址；
- 可选局部 OpenAPI；
- 流式能力说明；
- UI 嵌入能力说明；
- 鉴权方式说明。

这使得前端不再被迫把所有插件都看成“SDK 方法集合”。

### 9.3 MCP 插件：工具清单优先

MCP 插件对前端暴露的是：

- 工具列表；
- 参数 Schema；
- 工具描述；
- 执行入口；
- 返回格式与错误信息。

### 9.4 OpenAPI 聚合的边界

V2.1 明确规定：

- OpenAPI 聚合是 **API 插件增强能力**；
- 不是所有插件都必须进入聚合；
- 进入聚合的子应用必须做 schema 命名空间隔离。

聚合器最少需要解决：

- `paths` 前缀重写；
- `components/schemas` 前缀化；
- `$ref` 递归重写；
- 冲突检测与报错。

## 10. 进程模型与子进程治理（Process Model）

### 10.1 ProcessManager 是强制基建

所有需受网关生命周期托管的子进程，必须通过统一的 `ProcessManager` 启动。

禁止：

- 在插件中裸用 `subprocess.Popen` 启动受托管进程；
- 让插件自行决定是否清理子进程；
- 让 shutdown 逻辑分散在各插件中。

### 10.2 跨平台要求

ProcessManager 至少要解决：

- Windows：Job Object 或等价绑定；
- Unix：进程组 / `setsid` / `killpg`；
- 网关退出时统一回收；
- 异常退出后的最小残留控制。

### 10.3 冻结环境约束

打包环境下必须明确区分：

- 外部可执行文件；
- 当前主进程可执行文件；
- 多进程派生行为。

正式要求：

- 不得把自身冻结产物当成“python 解释器”再次执行脚本；
- 若使用 multiprocessing，入口必须 `freeze_support()`；
- 必须防止递归解压、自我派生、无限重启。

## 11. 构建与打包策略（Build & Packaging）

### 11.1 正式目标

构建系统的目标不是“打成单文件就算完成”，而是确保：

- 插件代码可被打包；
- 插件静态资源可被打包；
- 数据目录与资源目录行为一致；
- 冻结环境下子进程仍可被安全管理。

### 11.2 预构建扫描器是正式基建

在执行任何 Python 打包前，必须先运行预构建扫描器，负责：

- 扫描插件入口；
- 生成显式 import 或 hidden-import；
- 收集静态资源；
- 输出打包参数。

### 11.3 双轨打包策略

#### 轨道 A：Nuitka

适合：

- 依赖相对纯净；
- 追求启动速度；
- 希望提高代码保护能力。

#### 轨道 B：PyInstaller

适合：

- 依赖复杂；
- 动态反射多；
- 更看重兼容性而非极限性能。

### 11.4 打包边界说明

V2.1 不再宣传“单一打包策略可兼容所有第三方生态”，而是明确：

- Nuitka 不是所有复杂生态的默认答案；
- PyInstaller 也不是零成本兼容层；
- 打包策略必须与插件类型和依赖复杂度一起评估。

## 12. 分阶段落地策略（Delivery Phasing）

### 12.1 MVP（建议先实现）

MVP 建议只做：

- Tauri 宿主；
- Python 网关；
- API 插件；
- HTTP JSON 调用；
- 数据目录注入；
- ProcessManager；
- 优雅停机；
- 统一鉴权；
- Vite + React 前端。

MVP 明确**不做**：

- 服务型大插件；
- SSE；
- WebSocket 代理；
- OpenAPI 聚合多子应用；
- MCP 工具生态管理界面；
- 插件市场。

### 12.2 Phase 2

在 MVP 稳定后，再增加：

- OpenAPI 聚合器；
- MCP 桥接；
- 服务型插件 Manifest；
- 基础流式能力。

### 12.3 Phase 3

最后再演进：

- 复杂服务型插件代理；
- 插件启停管理界面；
- 插件安装/升级能力；
- 更细粒度权限隔离模型。

## 13. 典型接入场景（Developer Workflow）

### 13.1 场景 A：接入自研小型 API 工具

步骤：

1. 在 `plugins/` 下创建 API 插件目录；
2. 编写 `manifest.json` 与 `entry.py`；
3. 通过 `setup(context)` 注册 `APIRouter`；
4. 运行网关；
5. 生成或刷新前端 SDK；
6. 在前端增加 UI。

这是 V2.1 最优先支持的低风险路径。

### 13.2 场景 B：接入可适配的第三方项目

步骤：

1. 在插件目录中放置第三方项目；
2. 手写 `adapter.py`；
3. 显式注入数据目录、资源路径、必要配置；
4. 决定其属于：
   - 同进程 API 插件；
   - 服务型插件；
   - 独立进程代理插件；
5. 为其声明 Manifest；
6. 前端按 OpenAPI 或 Manifest 集成。

此类场景**不承诺零改造**。

### 13.3 场景 C：接入本地 MCP Server

步骤：

1. 注册 MCP server 定义；
2. 由网关通过 ProcessManager 启动；
3. 暴露工具列表与参数 Schema；
4. 前端渲染工具表单或操作面板；
5. 通过统一执行入口调用工具。

## 14. 风险清单（Risks）

V2.1 重点关注以下风险：

### 14.1 启动链路风险

- 结构化握手未实现；
- ready file 写入不原子；
- 端口已绑定但就绪状态未更新；
- 宿主超时策略不合理。

### 14.2 流式协议风险

- SSE 误用 `EventSource` 且未补鉴权；
- WebSocket token 泄露在可持久化日志中；
- 服务型插件流式协议与统一网关不兼容。

### 14.3 插件复杂度风险

- 错误把复杂第三方系统当成 API 插件；
- 依赖冲突被低估；
- 静态资源、状态目录、后台任务模型未摸清即接入。

### 14.4 打包风险

- hidden-import 不完整；
- 静态资源漏打；
- 冻结环境下进程递归派生；
- 打包测试只覆盖开发环境，不覆盖真实安装包环境。

## 15. 架构决策摘要（ADR Summary）

### ADR-001：宿主层选择 Tauri

- 保留 Tauri 作为桌面宿主；
- 但不把前端框架能力与桌面能力混淆。

### ADR-002：前端默认推荐改为 Vite + React

- Next.js 仍可支持；
- 但仅作为受约束备选方案。

### ADR-003：插件模型拆分为三类

- API 插件；
- 服务型插件；
- MCP 插件。

### ADR-004：OpenAPI 从“全插件统一基线”改为“API 插件优先契约”

- Manifest 成为服务型插件与 MCP 插件的重要补充契约。

### ADR-005：端口发现改为结构化握手

- 不再把 Uvicorn 日志解析作为正式机制。

### ADR-006：流式鉴权正式区分 HTTP / WS / SSE

- 默认以 `fetch + ReadableStream` 承载 SSE 类能力。

### ADR-007：全局 `os.chdir` 被正式禁止

- 数据目录通过显式注入完成；
- 资源路径通过框架 API 解析。

### ADR-008：ProcessManager 与优雅停机为强制基建

- 不允许插件绕过统一生命周期治理。

## 16. 最终结论（Conclusion）

V2.1 的核心价值，不在于把愿景缩小，而在于把愿景拆解成工程上可以验证、可以演进、可以治理复杂度的版本。

这份修订后的白皮书明确表达三件事：

1. **框架擅长什么**：托管本地能力、统一宿主、统一生命周期、统一契约；
2. **框架不擅长什么**：零成本吞并任意第三方大项目；
3. **框架应该如何落地**：先 MVP，再分阶段扩展插件类型与流式能力。

因此，V2.1 的正式主张是：

> 它不是“任意 Python 项目的一键桌面化魔法器”，而是一套面向本地能力集成、以约束复杂度为目标的桌面桥接基础设施。

在这个前提下，它仍然足够有野心，也更有机会真正落地。
