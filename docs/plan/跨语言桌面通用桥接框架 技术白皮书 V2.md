### 跨语言桌面通用桥接框架

### Universal Cross-Language Desktop Bridge

（面向资深工程师的技术白皮书 · V1）

---

## 1 引言（Introduction）

过去十年，桌面软件开发一直处在一种尴尬的撕裂状态：

- **前端生态**（React / Next.js / Tauri 等）拥有一流的 UI 表达力和跨平台能力，但运行环境受限于浏览器沙盒或轻量壳，**很难无缝复用本地 Python 能力**。  
- **Python 生态**则聚集了全球最繁荣的 AI、数据处理与自动化工具，但  
  - GUI 方案（如 Tkinter / PyQt / wxPython）在用户体验和前端工程化上明显落后  
  - 分发与环境管理成本极高，经常需要解释器 + 虚拟环境 + 复杂依赖安装流程  
  - 很难像现代桌面应用那样「一键安装、即开即用」。

与此同时，本地 AI 工具和自动化脚本呈现出 **高度碎片化** 的状态：  
每个项目都有自己的 CLI、配置方式、依赖环境和启动脚本。对于开发者和高级用户来说，「把一个优秀的 Python 项目变成可用的桌面工具」往往需要大量重复劳动——写 GUI、搞打包、解决跨平台兼容、维护安装说明。

本白皮书要解决的核心问题可以归结为一句话：

> **如何在不修改任何第三方 Python 项目源码的前提下，以最小的工程投入，把它们快速、安全地变成现代化的跨平台桌面级生产力工具？**

传统做法通常是为每个具体业务「分别写一套前后端 / 一套桌面壳」，这导致：

- 每个项目都在重复造轮子  
- 框架与工具之间缺乏统一的桥接抽象  
- 后续维护和演进成本成倍放大。

本框架尝试跳出「为某个业务写一套客户端」的思路，转而构建一个可复用的基础设施：

> 一个 **通用型桌面宿主 + 网关化 Python 底层** 的本地生态系统，让桌面壳只负责「托管」，让 Python 端只负责「能力」，中间通过统一的网关和协议适配，打通两大生态。

## 2 产品愿景与目标（Vision & Goals）

### 2.1 核心愿景

本项目的核心愿景可以用一句话概括：

> **让任何 Python 开源生态，在一天内化身为现代化的跨平台桌面级生产力工具。**

更具体地说：

- 对个人开发者：只要你能写出一个有价值的 Python 项目（脚本、API 服务、MCP Server 等），就可以在极短时间内，拥有一套 UI 体面、体验流畅、可跨平台分发的桌面应用壳。  
- 对技术团队：你们可以把各种内部工具、数据处理管线、AI Agent 系统挂接到统一的桌面入口，从而为团队构建一个 **本地能力中枢**。  
- 对开源生态：任意第三方 Python 项目，都可以以「插件」形式被挂载，而无需改动其原有工程结构和代码。

### 2.2 主要目标（In-Scope）

围绕上述愿景，本框架在 V1 版本中聚焦以下目标：

- **通用宿主，不带业务**  
  - 构建一个基于 Tauri V2 / Rust 的跨平台桌面宿主，**完全不包含业务逻辑**，只负责：  
    - 动态申请本地安全端口  
    - 启动并守护 Python 网关二进制（sidecar）  
    - 通过 IPC 向前端暴露网关端口信息
- **纯展示前端，不含底层逻辑**  
  - 使用 Next.js 作为纯展示层，禁止在前端直接编写底层业务逻辑。  
  - 所有后端能力通过 **OpenAPI → TypeScript SDK** 的方式，在 `src-client` 中被强类型化，然后由前端 UI 调用。
- **网关化 Python 能力中心**  
  - 在 Python 端使用 FastAPI 构建一个 **统一网关**：  
    - 负责挂载自研路由  
    - 以 ASGI 子应用的形式挂载第三方开源后端项目  
    - 通过 MCP 适配器，将本地 MCP Server 转换为 HTTP/SSE 接口
- **插件化能力模型**  
  - 所有业务能力（无论是自研工具还是第三方工程）都以「插件」形式存在于 `plugins/` 目录中，由网关统一扫描与挂载。  
  - 新增插件不需要修改宿主或前端架构，只需：放入目录 + 在网关注册。
- **双轨打包与物理固化**  
  - 提供两条官方推荐打包轨道：  
    - 轨道 A：Nuitka，将 Python 网关编译为 C++ 机器码，实现接近 0 冷启动与反编译防护  
    - 轨道 B：PyInstaller，自解压环境打包，兼容一切重度 AI 生态（如 LangChain / PyTorch）

### 2.3 非目标（Out-of-Scope / Non-Goals）

为了保持架构简洁，本框架在当前版本 **刻意不解决** 或 **暂不优先解决** 的问题包括（但不限于）：

- **远程或多机部署场景**  
  - 当前定位是「本地桌面 + 本地 Python 能力」，不直接针对分布式、Kubernetes 等集群环境。
- **多租户安全隔离 / RBAC / 权限系统**  
  - 当前假设单机单用户或信任边界在本地操作系统层面；细粒度权限控制可以作为后续演进方向。
- **云端服务编排与计费系统**  
  - 本框架专注于本地桥接与插件化，云端商业化能力不在 V1 设计范围。
- **统一数据模型 / 领域建模框架**  
  - 不试图为所有插件强行定义统一领域模型，而是通过 OpenAPI 和 MCP Schema 做「接口级统一」。
- **同进程内多套不兼容 Python 依赖的并存**  
  - 插件 A / 插件 B 形态属于**同解释器内的源码级整合**，所有插件共享同一个 Python 环境与 `requirements.txt`。框架**不负责**解决「多个依赖版本冲突的第三方项目硬塞进同一进程」的问题。若第三方项目存在严重依赖冲突（如 pydantic v1 vs v2、不同大版本的 langchain），应优先采用 **MCP 形态（插件 C）** 以独立进程运行，通过适配层接入网关。

通过明确这些 Non-Goals，可以避免在 V1 版本里引入过多复杂性，确保当前架构专注解决「本地桥接与插件化」这一核心问题。

## 3 总体架构总览（System Overview）

### 3.1 四象限本地生态系统

整个系统由四个互相解耦的象限构成，每个象限只负责清晰边界内的职责：

1. **宿主守护层（Tauri V2 / Rust）——「动态调度者」**
  - 以 Tauri 作为跨平台桌面外壳，通过 Rust 实现：  
    - 向操作系统申请一个当前绝对空闲的本地端口  
    - 以 **sidecar 进程** 形式唤醒 Python 编译后的二进制网关  
    - 负责监控子进程并在应用退出时清理僵尸进程
  - 不包含任何业务逻辑，只作为「安全、可控的本地执行环境托管者」。
2. **纯展示层（Next.js）——「动态渲染容器」**
  - 负责 UI 渲染和交互流程，严格禁止嵌入底层逻辑。  
  - 通过 Tauri IPC 向 Rust 宿主请求网关端口，并将其注入 `src-client` 中生成的 API SDK。  
  - 使用 `openapi-ts` 生成的 TypeScript SDK 进行强类型调用。  
  - 对于 MCP 插件，前端可以读取后端提供的工具 Manifest / Schema，**自动渲染对应的表单 UI**。
3. **网关与插件调度层（FastAPI / Python）——「万能插座引擎」**
  - 整个系统的「业务能力汇聚点」。  
  - 主程序不再直接写业务逻辑，而是：  
    - 暴露系统级接口（探活、插件列表、诊断信息）  
    - 提供 MCP 适配路由  
    - 将三类插件（自研小工具 / 第三方大项目 / MCP Server）统一挂载到一个 FastAPI 应用下。
4. **双轨物理固化层（Build System）——「资产保护伞」**
  - 通过 Nuitka 或 PyInstaller 对 Python 网关实现物理固化：  
    - Nuitka：AOT 编译为 C++ 机器码（适合纯净架构、核心私密算法）  
    - PyInstaller：自解压环境打包（适合依赖复杂 AI / 数据生态时的兼容性需求）
  - 打包产物统一丢入 `src-tauri/bin/`，由 Tauri 在构建时按不同平台打入安装包。

### 3.2 启动与运行时流程（Startup & Runtime Flow）

一个典型的应用启动流程如下：

1. 用户启动桌面应用（Tauri Shell）。
2. 宿主层通过 Tauri 路径 API 获取应用数据目录；**不**在 Rust 中预先 bind 端口（避免 TOCTOU，见 9.1）。
3. 宿主以 sidecar 方式启动 Python 网关二进制，传入 `**--port 0`**（由 Uvicorn 自行绑定空闲端口）与 `**--data-dir <绝对路径>**`（及可选的 `**--auth-token**`）。
4. Python 网关解析 `--data-dir` 并注入全局配置，以 `port=0` 启动 Uvicorn。Uvicorn 会向 stdout 打印原生日志（如 **`Uvicorn running on http://127.0.0.1:<port>`**）；宿主从该行解析真实端口并保存，再通过 `get_gateway_port` 开放给前端（因 uvicorn.run 为阻塞调用，无法在 Python 内再打印自定义 [READY]，见 9.1）。
5. Python 网关在启动过程中：
  - 加载系统路由  
  - 动态扫描并挂载所有插件（见 9.3）  
  - 初始化 MCP 适配器和相关资源。
6. 前端通过 Tauri `invoke('get_gateway_port')`（及可选的 `get_gateway_auth_token`）获取端口与鉴权信息，然后：
  - 将端口注入 `OpenAPI.BASE`  
  - 调用系统探活接口，确认网关可用  
  - 渲染首页和各类能力面板。
7. 用户在 UI 中点击按钮或填写表单，前端通过强类型 SDK（请求中附带鉴权 Token，若启用）调用网关统一接口，网关再将请求路由到对应插件或 MCP Server。

整个过程中：

- **Rust 宿主**掌握进程生命周期与端口信息  
- **Python 网关**负责能力挂载与协议适配  
- **Next.js 前端**只关心「有哪些能力可以调用」和「如何呈现给用户」，不感知插件内部实现。

### 3.3 数据流与控制流（Data & Control Flow）

从架构角度，可以把整个系统简化为三条主要流动：

- **控制流（Control Flow）**  
  - 宿主 → Python 网关：启动、端口传递、子进程守护  
  - 前端 → 宿主：通过 IPC 获取网关端口  
  - 前端 → Python 网关：通过 HTTP 调用统一 API（含 MCP 适配）
- **数据流（Data Flow）**  
  - HTTP 请求 / 响应在前端与网关之间流动  
  - 网关内部通过 Python 函数调用 / ASGI 派发，将数据交给不同插件  
  - MCP 场景下，HTTP 请求会在网关中被转换为 MCP JSON-RPC 请求，再通过 MCP SDK 发往本地 Server，最后再被封装回 HTTP 响应或 SSE 流。
- **构建与分发流（Build & Distribution Flow）**  
  - Python 网关通过 Nuitka / PyInstaller 编译为单一二进制  
  - Tauri 将前端打包为桌面应用，并将外部二进制通过 `externalBin` 打进安装包  
  - 最终用户获取到的只是一个统一的安装包，无需关心 Python 环境和依赖问题。

## 4 核心技术架构详解（Core Architecture）

### 4.1 宿主守护层（Rust / Tauri）：动态调度与进程托管

宿主层的职责可以概括为：**「申请端口 + 启动网关 + 提供端点查询」**。

典型实现结构如下（示意性代码片段）：

```rust
// src-tauri/src/main.rs
// Cargo.toml 需添加: reqwest = { version = "0.11", features = ["json"] }；tokio 通常已随 tauri 带入（用于 get_gateway_port 轮询与 sleep）

#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]

use tauri::Manager;
use tauri::WindowEvent;
use tauri_plugin_shell::ShellExt;
use tauri_plugin_shell::process::CommandEvent;
use std::sync::atomic::{AtomicU16, Ordering};
use std::time::Duration;

// 端口由 Uvicorn 启动时在 stdout 打印的原生日志中解析（见 9.1），避免 TOCTOU
static GATEWAY_PORT: AtomicU16 = AtomicU16::new(0);

/// 返回网关端口；若尚未就绪则轮询等待（避免前端「零毫秒竞态」拿到 0 导致探活失败，见正文哑弹二）。
#[tauri::command]
async fn get_gateway_port() -> Result<u16, String> {
    const MAX_RETRIES: u32 = 50; // 50 * 100ms = 5 秒超时
    for _ in 0..MAX_RETRIES {
        let port = GATEWAY_PORT.load(Ordering::SeqCst);
        if port != 0 {
            return Ok(port);
        }
        tokio::time::sleep(Duration::from_millis(100)).await;
    }
    Err("Timeout waiting for Python Gateway to bind a port".to_string())
}

fn main() {
    tauri::Builder::default()
        .plugin(tauri_plugin_shell::init())
        .setup(move |app| {
            let data_dir = tauri::api::path::app_data_dir(&app.config()).expect("app_data_dir");
            let data_dir = data_dir.to_string_lossy();
            let host_pid = std::process::id();
            // --port 0：由 Uvicorn 自行绑定；--host-pid：供 Python 看门狗轮询宿主存活，避免 stdin 管道「光速自杀」（见 10.9、11.2 附二）
            let sidecar_command = app.shell()
                .sidecar("bridge-gateway")
                .expect("Failed to create sidecar command")
                .args(["--port", "0", "--data-dir", &data_dir, "--host-pid", &host_pid.to_string()]);
            // 【必选】关闭 Python 子进程 stdout 缓冲，否则 Uvicorn 日志可能迟迟不 flush，宿主死等端口（见 9.5）
            let sidecar_command = sidecar_command.env("PYTHONUNBUFFERED", "1");

            let (mut rx, _child) = sidecar_command.spawn().expect("Failed to spawn Python Gateway");

            tauri::async_runtime::spawn(async move {
                while let Some(event) = rx.recv().await {
                    if let CommandEvent::Stdout(line) = event {
                        let s = String::from_utf8_lossy(&line);
                        println!("🌐 [Gateway]: {}", s);
                        // Uvicorn 启动时会打印 "Uvicorn running on http://127.0.0.1:54321"；解析该行中的端口（uvicorn.run 为阻塞调用，无法在 Python 内再打印 [READY]，见 9.1）
                        const PREFIX: &str = "http://127.0.0.1:";
                        if let Some(i) = s.find(PREFIX) {
                            let rest = s[i + PREFIX.len()..].split_whitespace().next().unwrap_or("");
                            if let Ok(port) = rest.parse::<u16>() {
                                GATEWAY_PORT.store(port, Ordering::SeqCst);
                            }
                        }
                    }
                }
            });

            Ok(())
        })
        // 【必选】优雅停机（致命陷阱四，见 10.8）：拦截关窗事件，先请求网关 shutdown 再退出，否则 Tauri 会直接杀 Sidecar，FastAPI lifespan 永不执行
        .on_window_event(|window, event| {
            if let WindowEvent::CloseRequested { api, .. } = event {
                api.prevent_close();

                let app_handle = window.app_handle().clone();
                tauri::async_runtime::spawn(async move {
                    let port = GATEWAY_PORT.load(Ordering::SeqCst);
                    if port != 0 {
                        let url = format!("http://127.0.0.1:{}/api/system/shutdown", port);
                        let client = reqwest::Client::new();
                        // 1. 发送优雅停机指令（Python 收到后设置 server.should_exit = True，Uvicorn 开始执行 lifespan shutdown）
                        let _ = client
                            .post(&url)
                            .timeout(Duration::from_secs(2))
                            .send()
                            .await;

                        // 【时序补丁】收到 200 后 Uvicorn 的 lifespan shutdown 已触发，须挂起 1～2 秒，
                        // 给予 Python 物理时间执行 ProcessManager.terminate_all() 与 DB 收尾；
                        // 若立刻 exit(0)，Tauri 会硬杀 Sidecar，shutdown 来不及完成（时序欺骗）。
                        tokio::time::sleep(Duration::from_millis(1500)).await;
                    }
                    // 2. 放行宿主退出，操作系统清理残局
                    app_handle.exit(0);
                });
            }
        })
        .invoke_handler(tauri::generate_handler![get_gateway_port])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

关键设计点：

- **反向握手消除 TOCTOU**：Rust 不预先 bind 端口；传 `--port 0` 给 Python，由 **Uvicorn** 在底层绑定空闲端口。Uvicorn 启动时为**同步阻塞**，无法在 Python 内再执行 `print("[READY]: ...")`，故宿主**不**依赖自定义握手行，而是从 **Uvicorn 原生日志** 解析端口：Uvicorn 会向 stdout 打印形如 `Uvicorn running on http://127.0.0.1:<port>`，宿主用正则（如 `http://127\.0\.0\.1:(\d+)`）提取端口并保存（见 9.1）。  
- **get_gateway_port 必须轮询等待（哑弹二：零毫秒竞态）**：WebView 渲染极快，而 Sidecar 从启动到 Uvicorn 打印端口通常需 0.5～3 秒。若 **get_gateway_port** 直接返回当前值，前端可能在端口仍为 0 时拿到 0，将 `OpenAPI.BASE` 设为 `http://127.0.0.1:0`，探活瞬间失败并呈现「系统网关连接失败」。因此 **get_gateway_port** 必须实现为**异步轮询**：在端口非 0 前每 100ms 轮询一次，最多等待约 5 秒；超时返回 **Err**，前端可提示超时；未超时则返回 **Ok(port)**。前端保持 **await invoke('get_gateway_port')** 即可优雅等待直至网关就绪。  
- **sidecar 进程模式**：Tauri 与 Python 网关作为两个进程运行，避免语言运行时互相污染。  
- **进程守护与日志转发**：通过 `tauri_plugin_shell` 监控子进程 stdout 并解析 Uvicorn 启动行。  
- **stdout 缓冲（深坑 9.5）**：启动 sidecar 时必须注入 **PYTHONUNBUFFERED=1**，否则 Python 子进程 stdout 带缓冲，Uvicorn 日志可能迟迟不发给宿主，导致端口解析死等、应用假死。  
- **关闭事件拦截与优雅停机（致命陷阱四，必选）**：**不得**让 Tauri 在用户点击「X」时默认直接退出（否则会连带杀死 Sidecar，FastAPI lifespan shutdown 永不执行，见 10.8）。必须在 **main.rs** 中监听 **tauri::WindowEvent::CloseRequested**，在处理器中：**1）阻止默认关闭**；**2）向 http://127.0.0.1:\<port\>/api/system/shutdown 发送 POST**（端口从 `GATEWAY_PORT` 取，若启用鉴权则携带 Token）；**3）等待响应或短超时（建议 2 秒）**；**4）收到 200 后必须再挂起约 1～2 秒**（时序补丁：200 表示 Python 已设置 should_exit，Uvicorn 的 lifespan shutdown 需物理时间；若立刻 app.exit(0)，Tauri 会硬杀 Sidecar，shutdown 来不及完成）；**5）再调用 app.exit(0)**，放行应用退出。这样 Python 网关才有时间执行 ProcessManager.terminate_all() 与 DB 收尾，再由 Tauri 清理，无僵尸进程与数据库损坏风险。  
- **宿主猝死与 --host-pid 轮询看门狗（极端盲区一）**：当宿主被任务管理器强杀或段错误崩溃时，CloseRequested 不触发，Sidecar 会残留为僵尸进程（见 10.9）。**禁止依赖 stdin 管道**：Tauri 侧易未正确维持 stdin pipe 或 Windows 下句柄为空，导致 Python 启动瞬间读到 EOF 而「光速自杀」。Rust 启动 Sidecar 时**必须**传入 **--host-pid \<当前进程 PID\>**（`std::process::id()`），Python 端以**轮询宿主进程存活**的方式实现看门狗，发现宿主死亡时设置 **server.should_exit = True** 触发优雅停机（见 11.2 附、11.2 附二）。

### 4.2 纯展示层（Next.js + OpenAPI SDK）

前端层是一个「只负责展示与交互」的 React / Next.js 应用，通过 Tauri IPC 获得网关端口，再通过自动生成的 SDK 完成调用。

简化示例：

```tsx
// app/page.tsx
'use client';

import { useEffect, useState } from 'react';
import { invoke } from '@tauri-apps/api/core';
// 这里的 SDK 包含系统 API、自研 API、第三方开源系统 API 的聚合
import { OpenAPI, SystemService, CustomToolsService } from '../src-client';

export default function UniversalDashboard() {
  const [status, setStatus] = useState<string>("初始化系统引擎...");

  useEffect(() => {
    async function initGateway() {
      try {
        // 1. 获取 Rust 动态分配的安全端口（内部轮询等待，最多约 5 秒，避免零毫秒竞态拿到 0）
        const dynamicPort = await invoke<number>('get_gateway_port');

        // 2. 注入全量 API Client 配置
        OpenAPI.BASE = `http://127.0.0.1:${dynamicPort}`;

        // 3. 探活验证
        const healthRes = await SystemService.healthCheck();
        setStatus(`网关就绪 (端口: ${dynamicPort}) | 状态: ${healthRes.status}`);
      } catch (e) {
        // get_gateway_port 超时或探活失败均会进入此处，可按 e 区分展示「等待网关超时」与「探活失败」
        setStatus("系统网关连接失败，请检查底层守护进程。");
      }
    }
    void initGateway();
  }, []);

  return (
    <div className="flex h-screen bg-[#f9fafb] text-gray-900 p-8 flex-col">
      {/* ... UI 细节略 ... */}
      <p className="text-sm font-mono text-gray-500 mt-2">OS Context: {status}</p>
    </div>
  );
}
```

设计要点：

- 前端不需要知道「有哪些具体插件」——只需要调用 `SystemService` / `CustomToolsService` 等 SDK 方法。  
- MCP 场景下，前端可以先通过 `/api/mcp/tools` 获取工具列表与参数 schema，再利用 `dynamic_forms/` 组件自动渲染表单。

### 4.3 网关与插件调度层（FastAPI / Python）

网关是整个系统的「能力总闸」。其主程序最重要的特点是：**自身不写业务，而是挂载业务**。**生产实现必须采用动态插件加载**：main.py 中**不手写**任何插件的 import 或 mount，而是通过扫描 `plugins/` 目录并执行插件契约（如 `setup(app)` 或 manifest 声明）完成挂载（见 9.3）。以下为逻辑示意（实际应以 importlib 动态扫描替代手写 import）。

示意代码（简化版，仅表达挂载形态）：

```python
# src-python/gateway_server/main.py
# 生产实现：通过 importlib 扫描 plugins/ 并调用各插件的 setup(app)，不在此处手写 import

import argparse
import uvicorn
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

from gateway_server.system_api.router import system_router
from gateway_server.core.mcp_adapter import mcp_router
# 动态加载：for p in scan_plugins(): p.setup(app)

app = FastAPI(
    title="Universal Desktop API Gateway",
    description="The central routing hub for all local plugins and MCP servers",
    version="4.0.0",
)

# 禁止 allow_origins=["*"]，否则恶意网页可跨域调用网关（见 9.2）。仅允许 Tauri 专属 Origin
app.add_middleware(
    CORSMiddleware,
    allow_origins=["tauri://localhost", "http://tauri.localhost"],  # 按 Tauri 版本调整
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# 1. 系统与 MCP 基础网关
app.include_router(system_router, prefix="/api/system")
app.include_router(mcp_router, prefix="/api/mcp")

# 2. 插件挂载：生产环境由 scan_plugins() + setup(app) 动态完成，此处仅示意
# 若启用 --auth-token，需在中间件中校验 Authorization: Bearer <Token>（见 9.2）

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--port", type=int, default=0)   # 0 表示由 Uvicorn 自动选端口
    parser.add_argument("--data-dir", type=str, required=True)
    parser.add_argument("--auth-token", type=str, default=None)
    args = parser.parse_args()

    # uvicorn.run() 为同步阻塞调用，无法在调用后再打印端口；宿主从 Uvicorn 原生日志 "Uvicorn running on http://127.0.0.1:<port>" 解析端口（见 4.1、9.1）
    uvicorn.run(app, host="127.0.0.1", port=args.port, log_level="warning", reload=False)

if __name__ == "__main__":
    import sys
    if getattr(sys, "frozen", False):
        import multiprocessing
        multiprocessing.freeze_support()  # 冻结环境下防无限重启（见 10.6、11.2）
    main()
```

三类插件形态简要说明：

- **插件 A（自研轻量工具）**  
  - 形式：一个或若干个 `APIRouter`，挂载在 `/api/custom_tools/...` 下  
  - 适合封装个人脚本、常用小工具、内部 API。
- **插件 B（第三方庞大开源项目）**  
  - 形式：通过 **适配器层（Adapter）** 挂载（见 4.3.1）；不鼓励粗暴的「原样 app.mount + Monkey Patch」。  
  - 例如某大型 AI Agent 系统、向量数据库管理面板等。
- **插件 C（MCP Server）**  
  - 形式：实现 MCP 规范的本地 Server，通过网关提供 HTTP/SSE 适配。  
  - 具体适配在 `mcp_adapter.py` 中完成。

#### 4.3.1 适配器模式（Adapter）与防腐层（Anti-corruption Layer）

本框架**正式承认**：「无缝挂载任意第三方项目且完全不改其代码」在工业实践中是**伪命题**。第三方项目往往深层次依赖相对路径、硬编码配置、无全局 Config 可 Patch，单纯依赖 **Monkey Patch** 脆弱且难以维护。  

**正式定义**：对**插件 B（第三方庞大开源项目）**，本框架要求引入 **适配器（Adapter）** 作为 **防腐层（Anti-corruption Layer）**。  

- 在 **plugins/plugin_b_vendor/** 目录下，开发者**必须**手写 **adapter.py**（或等价入口）。  
- 适配器职责：  
  - **实例化**第三方核心类或应用，**显式传入** `DATA_DIR`（及必要时 `__file__` / `sys._MEIPASS` 下的资源路径）作为数据/配置路径，而不是依赖进程 CWD 或 Monkey Patch。  
  - 将第三方能力**包装**为 `APIRouter` 或 ASGI 应用，再挂载到网关。  
- 这样网关与第三方之间的边界清晰：数据路径、静态资源路径均由适配器在构造时注入，不依赖运行时 Patch。  
- **Monkey Patch** 仅作为**补充**：当第三方暴露了单一、稳定的 Config 对象且路径逻辑较浅时，可在适配器内选择性使用；深层次或未暴露配置的第三方，**必须**通过适配器显式传参，不得依赖「改 CWD」或全局 Patch。  

插件契约中应约定：插件 B 的 **setup(app)** 由 **adapter.py** 实现，内部完成「加载第三方 + 注入 DATA_DIR/资源路径 + 挂载」。

**OpenAPI 与 Mount 的约束及修复（Schema 聚合）**  
FastAPI 底层（Starlette）规定：`app.mount()` 挂载的子应用，其 OpenAPI Schema **不会**合并到主应用的 `/openapi.json` 中。因此若直接对主网关的 `/openapi.json` 做 `openapi-ts` 生成，前端 SDK 将**只包含主应用路由**（如 `/api/system`、`/api/mcp`），不包含任何 Mount 子应用（如 `/vendor_ai_app`）的接口。  
本框架的**修复方案**：在网关层实现 **OpenAPI 聚合器**。对 `/openapi.json` 的请求进行拦截，在内存中拉取主应用的 schema，再逐一请求所有已挂载子应用各自根路径下的 OpenAPI，按挂载前缀重写 `paths`，并对 **components/schemas** 做 **命名空间重写**（见 9.4）：将每个子应用的 schema 名称统一加前缀（如 `PluginA_User`），并递归重写该子应用 paths 内所有 `$ref` 指向，再合并进全局 components，避免多插件同名 schema 覆盖导致类型错乱。最后返回统一后的全量 schema，供 `openapi-ts` 一次性生成强类型 SDK。

### 4.4 协议适配层：MCP → REST/SSE 桥接

由于 MCP Server 通常通过 `stdio` + JSON-RPC 提供能力，而浏览器（前端）无法直接以此方式调用本地进程，因此需要在网关中增加一个 MCP 适配层。

示意实现：

```python
# src-python/gateway_server/core/mcp_adapter.py

from fastapi import APIRouter
from pydantic import BaseModel
from typing import Dict, Any

mcp_router = APIRouter()

# 模拟动态扫描到的本地 MCP 插件
def load_local_mcp_servers():
    # 实际中会扫描 plugins/plugin_c_mcp/ 并实例化 MCP Server
    return [
        {"server_id": "sqlite_db", "tools": [{"name": "query_db", "description": "Execute SQL"}]},
        {"server_id": "local_fs", "tools": [{"name": "read_file", "description": "Read local file"}]},
    ]

@mcp_router.get("/tools")
async def get_all_mcp_tools():
    """向前端暴露当前系统挂载的所有 MCP 工具清单与 Schema"""
    return {"status": "success", "mcp_servers": load_local_mcp_servers()}

class MCPToolRequest(BaseModel):
    server_id: str
    tool_name: str
    arguments: Dict[str, Any]

@mcp_router.post("/execute")
async def execute_mcp_tool(req: MCPToolRequest):
    """前端统一的调用入口，这里将 HTTP JSON 转化为底层的 MCP JSON-RPC 调用"""
    # 伪代码：实际会调用底层 MCP SDK
    # result = await active_mcp_servers[req.server_id].call_tool(req.tool_name, req.arguments)
    result = f"Mock execution of {req.tool_name} on {req.server_id} with {req.arguments}"
    return {"status": "success", "result": result}
```

设计要点：

- 前端只需要调用 HTTP 接口 `/api/mcp/tools` 和 `/api/mcp/execute`，无需了解 MCP 细节。  
- 网关可以统一管理所有 MCP Server 的生命周期和日志，降低集成复杂度。

**MCP 子进程生命周期管理（防孤儿进程）**  
标准 MCP Server 通常以 stdio + JSON-RPC 的**独立进程**形式运行。网关若通过 `subprocess.Popen` 等拉起 MCP 子进程，当 Tauri 关闭并杀死 Python 网关时，未加清理的 MCP 子进程会变成孤儿进程常驻系统，多次开关应用将导致大量残留进程占用内存与句柄。  
本框架的**强制要求**：在网关内实现显式的进程树清理。  

- 网关启动 MCP Server 时，记录其 PID 及进程组（process group）。  
- 在 FastAPI 的 **lifespan** 或 **@app.on_event("shutdown")** 钩子中，统一清理所有由网关拉起的 MCP 子进程：Windows 下使用 Job Object 将子进程纳入同一 Job 随进程退出一并终止；Unix 下使用 `os.killpg` 等对进程组发送终止信号。  
- 确保「网关进程退出 ⇒ 所有 MCP 子进程同步退出」，无孤儿进程残留。  
- **前提**：上述 shutdown 钩子**只有在网关被优雅停机时才会执行**。若用户直接关闭 Tauri 窗口而宿主未做处理，Tauri 会直接杀死 Sidecar，lifespan/shutdown 不会运行（见 **10.8 终极死角**）。因此宿主**必须**拦截关闭事件并先请求 **POST /api/system/shutdown**，再退出（见 4.1、11.2 附）。

### 4.5 数据持久化路径注入（Data-Dir）

打包为单文件（尤其是 PyInstaller `--onefile`）后，进程往往在系统临时目录解压运行；插件或第三方若将数据写死在 `./data.db` 会导致落盘位置不可控或数据丢失。**禁止**在 main.py 级别执行 **os.chdir(DATA_DIR)**：单文件打包下 CWD 初始指向解压出的虚拟文件系统，第三方会用 `open("./config.yaml")` 等读**自带静态资源**，chdir 到 DATA_DIR 后这些读取会失败（见 10.5、11.4）。  
本框架的**设计**（与 11.4 一致）：  

- 宿主在启动 sidecar 时传入 **`--data-dir <绝对路径>`**，网关解析后**仅**将 **DATA_DIR** 写入 **core/config**，供全局访问。  
- **禁止全局 os.chdir**。数据落盘路径由各插件在 **setup()** 中通过**适配器层**或 **Monkey Patch** 精准覆写第三方 Config/路径变量为 `DATA_DIR`；静态资源读取**必须**通过框架提供的 **get_resource_path()**（见 9.6、11.5），不得在业务中硬写 `__file__` 或 `sys._MEIPASS` 导致开发/打包环境不一致。  
- **约定**：自研插件显式使用 `DATA_DIR`；加载 HTML/YAML/TXT 等资源时统一调用 **get_resource_path(relative_path)**；第三方项目通过 **adapter.py 防腐层** 在挂载前注入数据路径（见 4.3.1）。

### 4.6 双轨打包与物理固化（Build System）

**悖论一 + 致命陷阱三约束**：执行任一打包脚本之前，**必须先运行** **scripts/pre_build_scanner.py**：生成显式 import / --hidden-import，并生成 **--add-data**（PyInstaller）或 **--include-data-dir**（Nuitka）等静态资源参数并注入构建命令（见 10.1、10.7、11.1）。否则插件 .py 会漏打（ModuleNotFoundError）或静态资源丢失（带 UI/配置的插件必挂）。

为同时满足「启动速度 / 反编译防护」与「复杂依赖兼容」，设计了两条打包轨道。

#### 4.6.1 轨道 A：Nuitka 极速安全编译

适合依赖比较纯净、追求启动速度和代码保护的场景。


## 5 典型开发与使用场景（Developer Workflow）

### 5.1 场景一：挂载自研 Python 小工具（插件 A）

目标：将一个已有的 Python 脚本 / 小型 FastAPI 路由，变成一个在桌面应用中可点击调用的能力。

典型步骤：

1. 在 `src-python/gateway_server/plugins/` 下创建符合**插件契约**的目录（如实现 `setup(app)` 或提供 `manifest.json`）。
2. 在其中实现业务逻辑与 FastAPI `APIRouter`（或 Mount 子应用），在契约入口中完成 `app.include_router` / `app.mount`。
3. **无需修改 main.py**：网关启动时通过动态扫描 `plugins/` 自动加载（见 9.3、ADR-009）。
4. 启动 Python 网关，确保 OpenAPI 文档包含新路由（聚合器会合并并做 schema 命名空间重写）。
5. 运行 `pnpm gen:api`，由 `openapi-ts` 将所有接口聚合生成到 `src-client/`。
6. 在前端使用生成的 `CustomToolsService` 调用该能力，并为其设计 UI。

### 5.2 场景二：挂载第三方大型 AI 项目（插件 B）

目标：把某个 GitHub 上的复杂 AI 系统（如某 Agent 框架）带入桌面应用，通过**适配器防腐层**明确数据与资源路径，而非依赖「无缝挂载」或脆弱 Monkey Patch。

典型步骤：

1. 在 `plugins/plugin_b_vendor/` 下 **clone 第三方项目**，并**手写 adapter.py**（见 4.3.1）：在适配器内**实例化**第三方核心类或 app，**显式传入** `DATA_DIR` 作为数据路径、以及基于 `__file__` / `sys._MEIPASS` 的资源路径；再将实例包装为 `APIRouter` 或 ASGI 应用。  
2. 在 **setup(app)** 中挂载适配器暴露的路由或子应用（**不**鼓励直接 `app.mount(第三方_raw_app)` 且依赖 Monkey Patch）。  
3. **无需修改 main.py**：网关通过动态扫描与契约调用自动挂载（见 9.3）。  
4. 启动网关，确认子应用 OpenAPI 可访问。网关的 **OpenAPI 聚合器**会合并 schema 并做**命名空间重写**，再输出统一 `/openapi.json`。
4. 运行 `pnpm gen:api --input http://127.0.0.1:port/openapi.json`，即可得到包含主应用与所有 Mount 子应用接口的全量 TypeScript SDK，前端即可调用。

### 5.3 场景三：挂载本地 MCP Server（插件 C）

目标：将遵循 MCP 规范的本地工具纳入统一调用通道，并在前端通过动态表单承载。

典型步骤：

1. 在 `plugins/plugin_c_mcp/` 中放置 MCP Server 实现，如 `sqlite_mcp_server.py`。
2. 在 `mcp_adapter.py` 中实现对该 Server 的初始化和工具注册逻辑。
3. 网关启动后，`GET /api/mcp/tools` 即可返回当前系统中可用的 MCP 工具及其参数描述。
4. 前端请求该接口，使用 `dynamic_forms/` 组件根据 Schema 自动渲染表单。
5. 用户填写参数，前端调用 `POST /api/mcp/execute`，网关负责转换为 MCP JSON-RPC 调用并返回结果。

## 6 安全性与隔离性（Security & Isolation）

### 6.1 进程与权限边界

- **Tauri / Rust 宿主**  
  - 以受操作系统权限约束的普通 GUI 应用运行。  
  - 持有 Python 网关进程的生命周期控制权，但不直接操作业务数据。
- **Python 网关**  
  - 作为独立进程运行，负责真正的业务能力调度。  
  - 权限边界等同于运行该二进制的操作系统用户。
- **插件与 MCP Server**  
  - 可以运行在同一进程中（Python 插件）或各自独立进程中（某些 MCP 实现）。  
  - 建议对高风险插件进行额外的沙箱化或隔离（如容器 / 受限虚拟环境），这一点可以在未来演进中增强。

### 6.2 网络边界与本地 CORS/鉴权

- 网关默认仅监听 `127.0.0.1`，避免外网暴露。  
- 端口由 Uvicorn 绑定（`--port 0`）后通过反向握手上报宿主，避免 TOCTOU（见 9.1）。  
- **CORS**：**禁止** `allow_origins=["*"]`。必须将 `allow_origins` 严格限制为 Tauri 应用专属来源（如 `tauri://localhost`、`http://tauri.localhost`），防止恶意网页跨域调用网关（见 9.2）。  
- **可选鉴权**：建议宿主生成随机 Token 经 `--auth-token` 传入网关，网关校验请求头 `Authorization: Bearer <Token>`；前端通过 IPC 获取 Token 并在所有请求中附带。即使端口被探测，无 Token 也无法调用接口。  
- 若未来需要远程访问，应在上述基础上再增加显式配置与更强鉴权，而非默认放开。

### 6.3 插件信任模型

当前版本假设：

- **插件代码对使用者是「显式信任」的**：  
  - 即用户知道自己在挂载哪些第三方项目，对其行为承担责任。
- 框架侧更多关注 **运行稳定性与边界清晰**，而非提供完整的恶意插件防御体系。

## 9 生产级风险与设计约束（Production Risks & Constraints）

本节对四项易在生产环境暴露的「深坑」做**存在性分析**与**必选修复设计**，确保白皮书与实现一致、可落地且安全。

### 9.1 动态端口的竞态条件（TOCTOU）

**当前设计**：Rust 使用 `TcpListener::bind("127.0.0.1:0")` 取得一个空闲端口后，**释放该监听器**，再将端口通过 CLI 传给 Python，由 Python 去 bind 该端口。

**问题**：从 Rust 释放端口到 Python 成功 bind 之间存在数百毫秒「真空期」。此间操作系统可能将该端口分配给其他刚启动的服务，导致 Python 网关 bind 失败。这是典型的 **TOCTOU（Time-of-Check to Time-of-Use）** 竞态。

**结论**：**存在**。若实现按「先 Rust 占端口再释放再交给 Python」的方式，生产环境会出现间歇性启动失败。

**修复设计（反向握手 + 原生日志解析）**：  

- Rust 启动 Python 时传入 **`--port 0`**，由 Uvicorn 在系统底层自行寻找空闲端口并完成 bind。  
- **注意**：`uvicorn.run()` 为**同步阻塞**调用，程序会一直停在该行直到服务退出，无法在 Python 内再执行 `print("[READY]: ...")`，且难以在 run 内部拿到绑定端口。故**不**采用自定义握手行。  
- Rust 宿主在 sidecar 的 stdout 流中**解析 Uvicorn 原生日志**：Uvicorn 启动时会打印形如 **`Uvicorn running on http://127.0.0.1:<port>`**，宿主用正则或字符串解析提取端口，写入状态，再通过 `get_gateway_port` 开放给前端（见 4.1）。  
- 这样「占端口」与「使用端口」由 Uvicorn 在同一时刻完成，无真空期，消除 TOCTOU。  
- 实现时 4.1 中宿主仅解析 Uvicorn 启动行，4.3 中不依赖 Python 侧打印 [READY]。

### 9.2 本地 CORS/CSRF 与「形同虚设的沙盒」

**当前设计**：网关监听 `127.0.0.1`，并配置 `**allow_origins=["*"]`** 以便 Tauri 前端访问。

**问题**：任意来源均可跨域调用网关。用户若在**浏览器**中打开恶意网页，该页内 JavaScript 可向 `http://127.0.0.1:<port>` 发起请求（端口可通过探测或社会工程获取）。恶意页面即可冒充「本地应用」调用 MCP 工具或插件 API，实现本地读文件、删库等，等价于 **本地 CSRF/SSRF**。

**结论**：**存在**。`allow_origins=["*"]` 在生产环境不可接受。

**修复设计（二选一或叠加）**：  

- **Origin 严格限制（必选）**：**禁止** `allow_origins=["*"]`。将 CORS 的 `allow_origins` 严格限制为 **Tauri 应用专属来源**，例如 macOS 下 `tauri://localhost`、Windows 下 `http://tauri.localhost` 或 Tauri 文档中给出的当前版本专用 Origin。仅允许来自 Tauri WebView 的请求。  
- **动态 Token 鉴权（强烈建议）**：Rust 在启动 Python 网关时生成一个高熵随机 Token（如 32 字节 hex 或 UUID），通过 `**--auth-token <Token>`** 传入网关。网关在所有需要鉴权的路由或全局中间件中，校验请求头 `**Authorization: Bearer <Token>**`；无 Token 或 Token 不匹配则 401。前端在拿到端口时，同时通过 Tauri IPC 向 Rust 获取该 Token，并在生成的 SDK 或 fetch 封装中为所有请求自动附加该 Header。这样即使恶意页面猜到端口，没有 Token 也无法调用接口。

### 9.3 「伪」插件化与写死在代码里的导入

**当前设计**：示例中通过 `from gateway_server.plugins.plugin_a_custom.router import custom_router` 和 `app.include_router()` / `app.mount(...)` 挂载插件，且**每增加一个插件就需在 main.py 中新增 import 与挂载语句**。

**问题**：这仅是「按目录隔离」，**不是**真正的插件化。真正插件化应满足：**网关主干（main.py）不因新增/删除插件而改动**；新增插件只需放入约定目录并满足契约即可被加载。

**结论**：**存在**。若实现按当前示例的写法，每次拉取新第三方项目都要改 main.py，自动化与可维护性大打折扣。

**修复设计（动态扫描与插件契约）**：  

- 在 `gateway_server/main.py` 中采用 **importlib**（或等价机制）对 **plugins/** 目录进行**动态扫描**。  
- 定义**插件契约**：每个插件根目录下需提供以下之一：  
  - 约定文件名（如 `manifest.json`）声明类型与入口；或  
  - 统一入口函数 `**def setup(app: FastAPI) -> None`**（或 `register(app)`），由网关在发现插件目录后动态 import 并调用，将路由/Mount 挂载到主 `app`。
- 网关启动时：遍历 `plugins/` 下符合契约的子目录，逐个 import 并执行 `setup(app)`，**main.py 中不再包含任何插件名或手写 import**。  
- 这样新增插件 = 在 plugins 下新增符合契约的目录，无需改 main.py，实现真正的「热插拔」式挂载。

### 9.4 OpenAPI 聚合的 Schema 命名空间碰撞

**当前设计**：网关层实现 OpenAPI 聚合器，拉取主应用与各子应用 schema 后「深拷贝合并」paths 与 components。

**问题**：**未对 `components/schemas` 做命名空间隔离**。插件 A 与插件 B 可能都定义同名模型（如 `User`）。直接合并时后写入会覆盖先写入，导致生成的 TypeScript 类型与真实接口错位，强类型闭环被破坏。

**结论**：**存在**。聚合器若只做简单合并，多插件场景下必然出现 schema 重名覆盖。

**修复设计（Schema 命名空间重写）**：  

- 聚合器在合并**每个**子应用（或主应用外的每个挂载点）的 OpenAPI 时，必须对该子应用的 **components/schemas** 做**命名空间重写**。  
- 为每个挂载点分配固定前缀（如挂载路径规范化后的 `PluginA`、`VendorAiApp`）：  
  - 将该子应用下所有 schema 名称重命名为 `**<Prefix>_<原名称>`**（如 `PluginA_User`、`VendorAiApp_User`）。  
  - 递归处理该子应用 **paths** 下所有请求体/响应体中的 `**$ref: "#/components/schemas/xxx"`**，将 `xxx` 重写为 `**<Prefix>_xxx**`。
- 主应用自身的 schema 可保留原名或使用统一前缀（如 `Gateway_`），与插件前缀区分。  
- 合并时先重写、再并入全局 `components/schemas`，避免覆盖。这是实现「全量强类型 SDK」必须完成的工程步骤。

### 9.5 深坑：被 Python 吞噬的 stdout 缓冲区（The Buffered Stdout Trap）

**当前设计**：4.1 与 9.1 中，Rust 宿主通过监听 Python 子进程的 stdout，解析 `Uvicorn running on http://127.0.0.1:<port>` 获取动态端口。

**问题**：Python 的 **stdout 默认带缓冲**。当 Python 作为**子进程**（非 TTY）运行时，输出往往要攒到一定量（如几 KB）才 flush 给操作系统。Uvicorn 打印的那行带端口的日志可能被憋在进程内，Rust 一直收不到，导致宿主死等、前端一直「初始化中…」、应用假死。

**结论**：**存在**。不关闭子进程 stdout 缓冲时，端口解析依赖的日志可能延迟或无法到达宿主。

**修复设计（必选）**：Rust 在启动 Sidecar 时**必须**关闭 Python 侧 stdout 缓冲。  
- 在 **app.shell().sidecar(...)** 得到 command 后、**spawn() 之前**，为子进程注入环境变量 **`PYTHONUNBUFFERED=1`**（若 sidecar 为 Python 解释器启动的脚本）或等效方式，使 Python 运行时对 stdout 使用行缓冲或无缓冲。  
- 若 sidecar 为 PyInstaller/Nuitka 产出的可执行文件，其底层仍可能受 C 运行时缓冲影响；在可行前提下同样注入 **PYTHONUNBUFFERED=1**（部分运行时仍会尊重该变量），或确保 Uvicorn/日志库配置为立即 flush。  
- 实现见 4.1 与 11 节对 sidecar 环境变量的约定。

### 9.6 深坑：静态资源解析的「薛定谔路径」（sys._MEIPASS）

**当前设计**：4.5 与 11.4 要求静态资源读取保持基于 **`__file__`** 或 **sys._MEIPASS**。

**问题**：开发者在写 adapter 或插件时，若**硬写 sys._MEIPASS**，在**源码开发环境**下该属性不存在，直接报错；若**不写**，在**单文件打包环境**下资源在解压目录，基于 `__file__` 的路径指向错误位置，打包后运行报错。开发者不得不自己「猜」如何同时兼容两种环境。

**结论**：**存在**。缺乏统一轮子会导致适配器/插件在开发与打包间行为不一致。

**修复设计（必选）**：框架**必须**在 **core/** 提供现成的 **资源路径解析器（Resource Path Resolver）**。  
- 在 **core/config.py** 或 **core/path_resolver.py** 中提供统一函数，例如 **`get_resource_path(relative_path: str) -> str`**（或 `get_plugin_resource_base()` 返回插件资源根目录），内部实现**必须区分**：  
  - **冻结环境**（`getattr(sys, 'frozen', False)` 为 True）：**sys._MEIPASS 仅 PyInstaller 存在**，Nuitka 不设置该属性。故需 **hasattr(sys, '_MEIPASS')** 判断；若存在则用 **sys._MEIPASS**（轨道 B）；若不存在则用 **os.path.dirname(os.path.abspath(__file__))**（轨道 A Nuitka，其 __file__ 指向正确执行目录）。  
  - **源码环境**：以当前包或 **`__file__`** 所在目录为资源根，与 `relative_path` 拼接后返回。  
- **契约约定**：插件加载自身 HTML/YAML/TXT 等静态资源时，**统一**调用框架提供的 **get_resource_path(relative_path)**（或等价 API），不得在业务中硬写 `sys._MEIPASS` 或仅用 `__file__` 导致打包后或 Nuitka 轨道失效。  
- 实现见 11.5。

以上四项生产级风险与上述两项深坑（9.5 stdout 缓冲、9.6 资源路径）在生产实现中**必须**按修复设计落实；白皮书正文与 ADR 中与端口分配、CORS、插件挂载方式、OpenAPI 聚合、sidecar 环境、资源路径相关的描述，均以本节为准。下述**第十节（核心悖论）**与**第十一节（核心基建工具箱）**所涉悖论及内置脚本，同样为必选实现约束。

## 10 核心悖论与必选修复（Core Paradoxes & Mandatory Fixes）

本节对四组「设计目标之间的结构性冲突」做**存在性分析**与**必选破局方案**；对应落地手段统一纳入**第十一节核心基建工具箱**，框架必须提供现成实现，不得仅以文档「提倡」交由使用者自行处理。

### 10.1 悖论一：动态扫描 (importlib) 与物理固化 (Nuitka/PyInstaller) 的水火不容

**白皮书中的设计**：ADR-009 要求通过 importlib 动态扫描 `plugins/` 并加载插件，实现主干不手写 import；4.6 要求用 Nuitka 或 PyInstaller 将网关编译为单文件。

**矛盾**：Nuitka 与 PyInstaller 基于**静态分析**（AST/依赖图），只能识别**显式**的 `import xxx`。`importlib.import_module(dynamic_path)` 在编译时无法被追踪，打包工具不会把 `plugins/` 下被动态引用的模块打进产物。  
**后果**：打包「成功」，但运行 .exe 时立刻 **ModuleNotFoundError**，插件代码未被打包。

**结论**：**存在**。动态挂载与静态打包在当前形态下无法共存。

**破局方案（必选）**：引入 **预编译/钩子机制 (Pre-build Hook)**。  

- 在执行 `build_nuitka.sh` 或 `build_pyinstaller.sh` 的**第一步**，必须先运行一个 Python 脚本（见 11.1），扫描当前 `plugins/` 目录，根据插件契约识别所有需加载的模块。  
- 该脚本**自动生成**一个临时文件（如 `__generated_imports__.py`），内容为**显式 import**（如 `import gateway_server.plugins.plugin_a_custom as _p1`、`import ...`），或在 Nuitka/PyInstaller 调用时生成完整的 `--include-package` / `--hidden-import` 参数列表。  
- main.py 在启动时**隐式导入**该生成文件（或打包时已通过 hidden-import 将插件纳入），这样既保留「运行时通过契约发现插件」的语义，又满足打包工具的静态分析要求。  
- 不采用本方案的构建流程视为不完整，打包产物不可用于生产。

### 10.2 悖论二：「不改第三方源码」的承诺 vs 写死的数据路径

**白皮书中的设计**：核心愿景为「在不修改任何第三方 Python 项目源码的前提下」挂载；ADR-006 约定所有插件将持久化数据写入宿主注入的 `DATA_DIR`。

**矛盾**：第三方项目源码中常见硬编码路径（如 `db_path = "./local_data.db"`）。若**不改源码**，无法令其使用 `DATA_DIR`；若坚持「不改」，则其会继续写当前工作目录，打包后往往是只读安装目录或会随进程结束销毁的临时目录，导致数据丢失或权限错误。

**结论**：**存在**。愿景与数据路径约束在字面意义上冲突。

**破局方案（见 10.5 致命陷阱一）**：**禁止**在 main.py 级别使用全局 **os.chdir(DATA_DIR)**（在单文件打包下会破坏虚拟文件系统，见 10.5）。改为**优雅退让**：在文档中承认「无缝挂载且完全不改第三方代码」不现实。**优先**通过 **适配器（Adapter）防腐层**（见 4.3.1）：在 plugins/xxx 下手写 **adapter.py**，实例化第三方时**显式传入** DATA_DIR 与资源路径，再包装为路由或 ASGI 挂载。**仅当**第三方暴露单一、浅层 Config 时，方可在适配器内用 **Monkey Patch** 作为补充。DATA_DIR 仅写入 core/config；不在进程级改变 CWD（见 11.4）。

### 10.3 悖论三：动态 Token 鉴权中间件与 WebSocket/SSE 的冲突

**白皮书中的设计**：ADR-008 建议通过 `Authorization: Bearer <Token>` 校验防御 CSRF。

**矛盾**：浏览器标准 **WebSocket API 无法自定义 HTTP Header**。第三方插件 B（如大型 Agent 项目）内部大量使用 WebSocket（实时终端、大模型流式对话等）。若鉴权中间件**仅**检查 Header，所有 WebSocket 连接将无法携带 Token，导致 **401**，功能被直接拦截致死。

**结论**：**存在**。仅依赖 Header 的鉴权与 WebSocket 不兼容。

**破局方案（必选）**：鉴权逻辑必须**按协议分流**。  

- **HTTP 请求**：继续使用 `Authorization: Bearer <Token>`。  
- **WebSocket 请求**（`scope["type"] == "websocket"`）：**必须**支持从以下至少一种方式提取 Token 并校验：  
  - **URL Query**：如 `?token=xxx` 或 `?access_token=xxx`；  
  - **WebSocket Subprotocols**：在建立连接时通过 Subprotocol 携带 Token（若前端与第三方均支持）。
- 框架提供的鉴权中间件（见 11.3）必须内置上述分流与 WebSocket 兼容逻辑，不得仅实现「全请求检查 Header」。

### 10.4 悖论四：「进程清理」的规范与缺少现成轮子

**白皮书中的设计**：ADR-005 要求网关在 shutdown 时对 MCP 子进程做 Windows Job Object / Unix killpg 清理，防止孤儿进程。

**矛盾**：在 Python 中正确实现 Windows Job Object 需调用 `ctypes.windll` 或引入 pywin32；若白皮书仅「提倡」而不提供实现，绝大多数使用者会偷懒只写 `subprocess.Popen`，不做进程树绑定，最终软件仍会留下大量僵尸进程。

**结论**：**存在**。规范与可落地性脱节。

**破局方案（必选）**：框架作为**基础设施**，**必须提供现成轮子**，不得将底层 OS 进程隔离交给使用者自行实现。  

- 在 **core/** 下内置 **ProcessManager** 单例（见 11.2），封装：  
  - 子进程启动时与当前进程的**进程树绑定**（Windows Job Object / Posix setsid 等）；  
  - 网关 shutdown 时**统一回收**所有经 ProcessManager 拉起的子进程。
- 插件/MCP 适配器**只能**通过 `ProcessManager.spawn(cmd)` 等统一接口启动子进程，由框架全自动接管生命周期；禁止在业务代码中直接使用裸 `subprocess.Popen` 启动需随网关退出的进程。

### 10.5 致命陷阱一：os.chdir(DATA_DIR) 会摧毁 PyInstaller 的虚拟文件系统

**V2 中的设计**：在 10.2 与 11.4 中规定在网关启动最早期执行 **os.chdir(DATA_DIR)**，挟持整个进程工作目录，使第三方相对路径落盘到安全目录。

**问题**：在**源码运行**下可行，但在 **PyInstaller --onefile** 下是**核弹**。单文件打包启动时会在临时目录（如 `/tmp/_MEIxxxxx`）解压虚拟环境，CWD  initially 指向该处，第三方不仅写数据，还会用 **open("./config.yaml")**、**open("./prompt.txt")** 等读**自带静态资源**。一旦执行 **os.chdir(DATA_DIR)**，CWD 被切到 AppData，此时 `open("./config.yaml")` 会在 AppData 里找，找不到 → 插件崩溃。

**结论**：**存在**。全局 chdir 与单文件打包的虚拟文件系统不兼容。

**破局方案（必选）**：**废弃全局 os.chdir**。  
- **禁止**在 main.py 级别改变整个进程的 CWD。  
- **优雅退让**：在文档中承认「完全不改第三方代码」在 Python 生态下不成立。在插件 **setup()** 中通过 **Monkey Patch**（patch 或覆写第三方 Config/路径变量）**精准**将**数据落盘路径**改为 `DATA_DIR`，**保留**其基于 **`__file__`** 或 **sys._MEIPASS** 的静态资源读取路径（见 11.4）。

### 10.6 致命陷阱二：ProcessManager 的「无限重启地狱」

**V2 中的设计**：在 10.4 与 11.2 中由 ProcessManager 封装子进程启动（如 MCP Server）与清理。

**问题**：单文件打包后没有独立 `python` 解释器，只有 **bridge-gateway.exe**。若在代码中**误用自身可执行文件**派生子进程（或 multiprocessing 使用不当），PyInstaller 会再次解压环境并执行主逻辑，触发**无限递归解压**，内存与 CPU 爆满、应用卡死。

**结论**：**存在**。冻结环境下子进程启动必须严格约束。

**破局方案（必选）**：在 **11.2 core/process_manager.py** 规范中**强制增加「冻结环境（Frozen Environment）检测」**。  
- 若 **getattr(sys, 'frozen', False)** 为 True，说明当前为打包环境。  
- **ProcessManager.spawn** 启动子进程时：**禁止**将主进程可执行文件（sys.executable）作为「python 解释器」去执行脚本；必须明确区分「调用外部 MCP 可执行文件」与「误用自身 exe」。  
- 正确设置环境变量（如 **LD_LIBRARY_PATH**、**sys._MEIPASS** 等），避免子进程再次触发解压。  
- 若使用 **multiprocessing**，主入口**必须**在最早处调用 **multiprocessing.freeze_support()**。  
- 文档与实现中明确上述约束，避免「无限重启地狱」（见 11.2）。

### 10.7 致命陷阱三：预编译扫描器漏掉「静态资源黑洞」

**V2 中的设计**：在 10.1 与 11.1 中 pre_build_scanner.py 仅生成 **__generated_imports__.py** 或 --hidden-import，解决 .py 代码被打包的问题。

**问题**：扫描器只扫 **Python 代码 (.py)**。插件 B 目录下常有 **templates/index.html**、**static/style.css**、**data/stopwords.txt**、**config.yaml** 等。显式 import 只能让打包工具把 .py 打进产物，**非 .py 静态资源**不会被纳入，打包后直接丢失，涉及 UI 或配置的插件必挂。

**结论**：**存在**。仅生成 import 不足以支持带静态资源的插件。

**破局方案（必选）**：**pre_build_scanner.py 职责升级**。  
- 扫描 **plugins/** 时，除识别 .py 模块外，**必须**识别并收集**非 .py 静态资源**（按契约或约定目录如 templates/、static/、data/、*.yaml 等）。  
- **自动生成**并注入构建脚本：  
  - **PyInstaller**：**--add-data** 或 **--add-datas** 参数列表；  
  - **Nuitka**：**--include-data-dir** 或等价参数。  
- 构建脚本在执行 Nuitka/PyInstaller 时**必须**使用扫描器输出的静态资源参数，否则带 UI/配置的第三方插件打包后必挂（见 11.1）。

### 10.8 终极死角：Tauri 粗暴杀进程与 FastAPI shutdown 的脱节

**白皮书中的设计**：在 4.4 与 11.2 中规定在 FastAPI 的 **lifespan** 或 **@app.on_event("shutdown")** 钩子中调用 **ProcessManager.terminate_all()**，统一清理 MCP 子进程并做收尾工作。

**残酷的物理现实**：当用户点击 Tauri 桌面应用的「X」关闭窗口时，Tauri 默认会**连带杀死 Sidecar 进程**。在 Windows 下，Tauri 底层使用的是 **TerminateProcess** API（等价于 Unix 下的 `kill -9` / SIGKILL）。这种操作系统级别的「硬杀」会**瞬间抹除进程**，完全绕过：

- Python 的 **atexit** 机制  
- **Signal** 信号处理（SIGTERM 等）  
- **FastAPI 的 lifespan / shutdown 钩子**

因此 **ProcessManager.terminate_all() 永远不会被执行**。若 Windows Job Object 封装不够底层（或仅用纯 Python 模拟），MCP Server 仍会变成僵尸进程；若第三方大模型插件正在写 SQLite 或 WAL 日志，被硬杀后极易导致 **Database Corruption**。

**结论**：**存在**。仅依赖「进程退出时自然走 lifespan shutdown」在 Tauri + Sidecar 模型下不成立，必须把**关闭主动权**从操作系统的屠刀下抢回来。

**破局方案（必选）**：引入 **「优雅停机（Graceful Shutdown）握手协议」**，作为框架生命周期管理的强制契约。

1. **Python 网关层增加「自尽」接口**  
   - 在 **gateway_server/system_api/router.py** 中**必须**强制暴露 **POST /api/system/shutdown**。  
   - 收到该请求时，网关**不得**立刻 `sys.exit()`，而应**优雅地向 Uvicorn 发送停止信号**（例如设置 Server 的 `should_exit` 或抛出约定异常），让 FastAPI 有机会走完完整 lifespan 拆卸流程（关闭 DB 连接、执行 **ProcessManager.terminate_all()**、释放资源）。

2. **Rust 宿主层拦截关闭事件**  
   - 在 **src-tauri/src/main.rs** 中，**绝不能让 Tauri 默认直接干掉应用**。必须拦截窗口的 **tauri::WindowEvent::CloseRequested**。  
   - 拦截到关闭请求后，Rust **阻止窗口立刻关闭**。  
   - 向 **http://127.0.0.1:\<port\>/api/system/shutdown** 发送 HTTP POST（若启用鉴权则携带 Token）。  
   - **等待**该请求返回成功，或等待一个**极短超时**（建议 2 秒）。  
   - 等待结束后，Rust 再放行 **app.exit()**；此时 Python 网关已优雅停机，Tauri 再清理残骸，安全无虞。

实现细节与契约见 **4.1（Rust 关闭事件拦截）**、**11.2（ProcessManager）** 及 **11.2 附（system_api 优雅停机接口）**。

### 10.9 极端盲区一：宿主「猝死」导致的僵尸网关（The Hard-Crash Zombie Trap）

**前提**：10.8 的优雅停机握手协议依赖 **Tauri 被用户正常关闭**（点击 X 或 Cmd+Q），从而触发 **WindowEvent::CloseRequested**。

**残酷的物理现实**：若宿主进程被**非正常终止**——例如用户在任务管理器中点击「结束任务」、或 Tauri 因段错误（Segfault）崩溃——**CloseRequested 根本不会触发**。Tauri 瞬间死亡，而 **tauri_plugin_shell** 拉起的 Python Sidecar 进程会**残留在后台**，成为僵尸网关。用户再次启动应用会再拉起一个新的 Python 进程，长期运行将堆积大量占用内存的僵尸进程。

**结论**：**存在**。仅靠「关窗握手」无法覆盖「宿主猝死」场景，必须让 Sidecar 在**检测到宿主消失**时自行触发优雅停机。

**为何废弃 stdin 管道看门狗（光速自杀陷阱）**：若以 **sys.stdin.read()** 阻塞等待「管道断裂」检测宿主死亡，在 Tauri 下 **tauri_plugin_shell** 极易未正确维持 stdin 管道（或 Windows 下 Sidecar 的 stdin 句柄为空）。Python 网关一启动即读到 **EOF** 或触发 **AttributeError: 'NoneType'**，看门狗误判宿主已死，在启动后 **1 毫秒内光速自杀**，前端永远显示连接失败。故**禁止**依赖 stdin，改用 **--host-pid 轮询**。

**破局方案（必选）**：**传递宿主 PID + 轮询存活**，工业级、零管道依赖：

1. **Rust 端**：启动 Sidecar 时传入 **--host-pid \<当前进程 PID\>**（`std::process::id()`），不依赖 stdin 管道状态。
2. **Python 端**：网关**编程式启动 Uvicorn**（见下），将 **Server 实例** 注入 **app.state.server**。在 **server.run() 之前** 启动守护线程 **\_host_watchdog(host_pid, server)**：用跨平台 API **轮询宿主进程是否存活**（Windows：`OpenProcess` + `GetExitCodeProcess` 判 STILL_ACTIVE；Unix：`os.kill(pid, 0)`）。当发现宿主已死，设置 **server.should_exit = True**，Uvicorn 主循环退出并执行 lifespan shutdown，**ProcessManager.terminate_all()** 得以执行。

实现见 **4.1（Rust 传 --host-pid）**、**11.2 附（should_exit 停机）**、**11.2 附二（main.py 编程式 Uvicorn + PID 看门狗）**。

## 11 核心基建工具箱 (Core Toolkit) 规范

本框架**必须**自带以下底层脚本与模块，作为「与悖论及生产风险一一对应的」不可或缺基建。实现时不得省略或降级为「示例」；使用者仅能在此规范之上扩展，不得替换或绕过核心行为。

### 11.1 scripts/pre_build_scanner.py（解决悖论一 + 致命陷阱三：动态扫描、静态打包与静态资源）

- **职责**：在 **Nuitka/PyInstaller 构建前** 被 **build_nuitka.sh / build_pyinstaller.sh 第一步调用**。  
- **行为（Python 模块）**：  
  - 扫描当前 `gateway_server/plugins/` 目录，根据插件契约（manifest 或 `setup(app)` 所在包）识别所有需参与运行时加载的 Python 模块/包。  
  - 生成临时文件 **__generated_imports__.py**（或等价），内容为**显式 import** 列表，或直接输出 **--hidden-import / --include-package** 参数列表供构建脚本拼接。  
- **行为（静态资源，致命陷阱三）**：  
  - 扫描 **plugins/** 时，**必须**同时识别**非 .py 文件**（如 templates/、static/、data/、*.yaml、*.txt、*.html 等）。  
  - **自动生成** **--add-data**（PyInstaller）或 **--include-data-dir**（Nuitka）参数列表，并在执行打包脚本时**注入**到 Nuitka/PyInstaller 调用中。否则带 UI 或配置文件的第三方插件打包后静态资源丢失，运行必挂（见 10.7）。  
- **集成**：main.py 在应用启动时 import 生成文件（若存在），或构建脚本将 hidden-import 与**静态资源参数**一并注入。  
- **约定**：每次新增/删除插件或插件内静态资源后，**必须**重新执行预编译步骤再打包。

### 11.2 core/process_manager.py（解决悖论四 + 致命陷阱二：进程树清理与冻结环境兼容）

- **职责**：提供**跨平台**的「子进程与当前进程绑定、随进程退出一并终止」能力，且**在单文件打包环境下安全**。  
- **行为**：  
  - 对外暴露单例或工厂，如 **ProcessManager.spawn(cmd, args, cwd=..., env=...)**，返回进程句柄或封装对象。  
  - 内部在 **Windows** 使用 **Job Object**（ctypes 或 pywin32）；在 **Unix** 使用 **setsid** 等建立进程组。  
  - 在 FastAPI **lifespan 的 shutdown** 中调用 **ProcessManager.terminate_all()**，统一终止所有经该 Manager 拉起的子进程。  
- **冻结环境（致命陷阱二）**：  
  - **必须**在模块内做 **getattr(sys, 'frozen', False)** 检测。当为**打包环境**时：  
    - **禁止**将**当前进程可执行文件**（sys.executable）当作「python 解释器」去执行子脚本，否则会触发 PyInstaller 再次解压、无限递归，内存/CPU 爆满（见 10.6）。  
    - 启动子进程时正确设置 **LD_LIBRARY_PATH**、**sys._MEIPASS** 等环境变量，避免子进程误触解压。  
    - 若使用 **multiprocessing**，主入口**必须**在最早处调用 **multiprocessing.freeze_support()**。  
  - 文档与代码注释中明确「冻结环境下仅允许启动外部 MCP 可执行文件等，禁止误用自身 exe」。  
- **约束**：MCP 适配器及需随网关退出的子进程**必须**通过 ProcessManager 启动，禁止裸 `subprocess.Popen`（除非明确声明为不随网关退出的常驻服务）。
- **优雅停机依赖（致命陷阱四，见 10.8）**：**ProcessManager.terminate_all()** 只有在网关**优雅停机**时才会被调用。优雅停机由宿主在用户关闭窗口时先请求 **POST /api/system/shutdown** 触发，Uvicorn 正常退出后执行 lifespan shutdown，进而执行 terminate_all()。若网关被 Tauri 或操作系统直接杀进程，shutdown 钩子**不会**执行，子进程与资源清理均无法保证。因此必须配合 **system_api 优雅停机接口** 与 **Rust 关闭事件拦截** 使用（见 4.1、11.2 附）。

**11.2 附：gateway_server/system_api 系统接口与优雅停机契约（解决致命陷阱四 + 终极死角二：Windows 信号黑洞）**

- **位置**：**gateway_server/system_api/router.py**（与探活、插件列表等系统接口同处）。
- **强制接口**：**必须**暴露 **POST /api/system/shutdown**。  
  - 收到该请求时，网关**不得**直接 **sys.exit()** 或 **os._exit()**（否则会跳过 lifespan shutdown）。  
  - **禁止依赖信号**：在 Windows 无控制台进程（如 Tauri Sidecar 带 CREATE_NO_WINDOW）下，**signal.raise_signal(signal.SIGINT)** 可能触发 C 运行时异常，被系统当作 Unhandled Exception **瞬间强杀（Exit Code 3）**，Uvicorn 与 lifespan 均不执行（终极死角二）。**必须**采用 **Uvicorn 内置的「自杀开关」**：**server.should_exit = True**。网关需**编程式启动 Uvicorn**（见 11.2 附二），将 **Server 实例** 存入 **app.state.server**，本接口内直接设置 **request.app.state.server.should_exit = True**，Uvicorn 停止接收新请求并**完整走完 lifespan shutdown**（ProcessManager.terminate_all()、关闭 DB 等）。  
  - 若启用鉴权，该接口**必须**校验与其它系统 API 一致的 Token（Header 或 query），避免被未授权方误触。
- **与宿主配合**：Rust 宿主在用户点击关闭时**必须先**调用本接口并等待（或短超时），再执行 **app.exit()**（见 4.1）。

**标准实现参考**（降维打击：不依赖 signal，直接拨动 Uvicorn 生命周期开关；需配合 main.py 的编程式启动与 app.state.server 注入）：

```python
# gateway_server/system_api/router.py
# 前提：main.py 已编程式创建 uvicorn.Server 并注入 app.state.server（见 11.2 附二）

from fastapi import APIRouter, Request

system_router = APIRouter()

@system_router.post("/shutdown")
async def shutdown_gateway(request: Request):
    """Rust 宿主在关闭窗口前调用。不依赖 signal，直接拨动 Uvicorn 的 should_exit，必定走完 lifespan shutdown。"""
    server = getattr(request.app.state, "server", None)
    if server is not None:
        server.should_exit = True
    return {"status": "shutting_down", "message": "Executing pure graceful shutdown..."}
```

说明：路由挂载时若使用前缀（如 `app.include_router(system_router, prefix="/api/system")`），则 **POST /shutdown** 即 **POST /api/system/shutdown**。设置 **should_exit** 后 Uvicorn 主循环退出，lifespan 的 shutdown 阶段执行 **ProcessManager.terminate_all()** 等收尾逻辑。

**11.2 附二：main.py 编程式 Uvicorn + 宿主猝死看门狗（--host-pid 轮询，解决极端盲区一 + 终极死角二）**

- **场景**：宿主猝死时 CloseRequested 不触发，Sidecar 会残留为僵尸（见 10.9）；且 Windows 无控制台下 signal 易导致强杀（终极死角二）。  
- **职责**：**1）** 网关**必须**以**编程式**创建 **uvicorn.Server**，将实例注入 **app.state.server**，供 **/shutdown** 路由与看门狗使用 **server.should_exit = True** 触发优雅停机，**禁止**依赖 **signal**。**2）** 接收 Rust 传入的 **--host-pid**，在 **server.run() 之前** 启动守护线程 **\_host_watchdog(host_pid, server)**：跨平台轮询宿主进程是否存活，发现宿主已死时设置 **server.should_exit = True**。  
- **前提**：Rust 启动 Sidecar 时**必须**传入 **--host-pid**（见 4.1）。

**标准实现参考**（main.py 入口：编程式 Uvicorn + PID 看门狗）：

```python
# gateway_server/main.py（核心片段：参数、Server 创建、看门狗、阻塞启动）

import argparse
import platform
import threading
import time
import os
import ctypes
import uvicorn
from fastapi import FastAPI

# app 的创建、插件挂载等略；以下在 main() 中、在创建 Config/Server 之前完成
def _host_watchdog(host_pid: int, server: uvicorn.Server) -> None:
    """跨平台轮询宿主存活；宿主死后拨动 should_exit，触发 Uvicorn 优雅停机与 lifespan shutdown。"""

    def is_alive(pid: int) -> bool:
        if platform.system() == "Windows":
            # 0x1000 = PROCESS_QUERY_LIMITED_INFORMATION；259 = STILL_ACTIVE
            handle = ctypes.windll.kernel32.OpenProcess(0x1000, False, pid)
            if not handle:
                return False
            exit_code = ctypes.c_ulong()
            ctypes.windll.kernel32.GetExitCodeProcess(handle, ctypes.byref(exit_code))
            ctypes.windll.kernel32.CloseHandle(handle)
            return exit_code.value == 259
        try:
            os.kill(pid, 0)
            return True
        except ProcessLookupError:
            return False
        except OSError:
            return True

    while True:
        if not is_alive(host_pid):
            server.should_exit = True
            break
        time.sleep(2)

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--port", type=int, default=0)
    parser.add_argument("--data-dir", type=str, required=True)
    parser.add_argument("--host-pid", type=int, required=True)
    args = parser.parse_args()

    # 解析 DATA_DIR、挂载插件，得到 app（略）
    app: FastAPI = ...

    config = uvicorn.Config(app, host="127.0.0.1", port=args.port, log_level="warning", reload=False)
    server = uvicorn.Server(config)
    app.state.server = server

    threading.Thread(target=_host_watchdog, args=(args.host_pid, server), daemon=True).start()
    server.run()

if __name__ == "__main__":
    import sys
    if getattr(sys, "frozen", False):
        import multiprocessing
        multiprocessing.freeze_support()
    main()
```

### 11.3 core/middlewares/auth.py（解决悖论三：WebSocket 兼容鉴权）

- **职责**：在启用 `--auth-token` 时，对请求做鉴权校验，且**不阻断** WebSocket/SSE。  
- **行为**：  
  - **HTTP**：从 Header `Authorization: Bearer <Token>` 提取 Token，与网关持有的 Token 比对；不通过则 401。  
  - **WebSocket**（`scope["type"] == "websocket"`）：**必须**支持从 **URL query**（如 `?token=xxx`）或 **Subprotocols** 中提取 Token 并校验；不通过则拒绝连接（如 403）。  
  - 中间件需在 ASGI 层根据 `scope["type"]` 分流，不得对 WebSocket 仅做 Header 检查。
- **配置**：从网关全局 config 读取 `AUTH_TOKEN`（由 `--auth-token` 注入）；若未设置则中间件可 no-op 或仅做 CORS。

### 11.4 数据目录注入（解决悖论二 + 致命陷阱一：禁止全局 chdir，改用精准 Monkey Patch）

- **职责**：将 `DATA_DIR` 注入全局配置，供插件与适配器使用；**不**通过改变进程 CWD 挟持路径。  
- **执行节点**：在 **gateway_server/main.py** 中，在 **argparse 解析 `--data-dir` 之后**，**仅**将 **DATA_DIR** 写入 **core/config** 或全局可访问处；**禁止**在 main.py 级别执行 **os.chdir(DATA_DIR)**（单文件打包下会破坏 PyInstaller 虚拟文件系统，第三方用 `open("./config.yaml")` 读自带资源时会失败，见 10.5）。  
- **路径落盘**：在 **gateway_server/plugins/xxx/setup()** 中，通过 **Monkey Patch** 或适配器显式传参将**数据落盘路径**改为 `DATA_DIR`；静态资源读取**必须**通过框架提供的 **get_resource_path()**（见 11.5），不得硬写 `__file__` 或 `sys._MEIPASS`（见 9.6）。  
- **文档与契约**：在插件契约中明确「DATA_DIR 由 config 提供；禁止依赖进程 CWD；数据路径须在 setup() 中通过适配器/Patch 注入；加载 HTML/YAML/TXT 等资源时**统一**调用 **get_resource_path(relative_path)**」（见 11.5）。

### 11.5 资源路径解析器（解决深坑 9.6：薛定谔路径 sys._MEIPASS；极端盲区二：Nuitka 无 _MEIPASS）

- **职责**：为插件与适配器提供**统一**的静态资源根路径，同时兼容**源码环境**、**PyInstaller 打包**与 **Nuitka 打包**，避免开发者硬写 `sys._MEIPASS` 或仅用 `__file__` 导致某一侧报错。  
- **位置与 API**：在 **core/config.py** 或 **core/path_resolver.py** 中提供现成函数，例如：  
  - **`get_resource_path(relative_path: str) -> str`**：返回「资源根目录 + relative_path」的绝对路径；  
  - 或 **`get_plugin_resource_base() -> str`**：返回当前插件（调用方）的资源根目录，由调用方再与相对路径拼接。  
- **行为（必须区分 PyInstaller 与 Nuitka）**：  
  - **sys._MEIPASS 是 PyInstaller 专属**；Nuitka 虽设 **sys.frozen=True**，但**不设置 sys._MEIPASS**。若冻结环境下仅用 `sys._MEIPASS`，Nuitka 轨道会 AttributeError 或资源寻址错误。  
  - **冻结环境**（`getattr(sys, 'frozen', False)` 为 True）：若 **hasattr(sys, '_MEIPASS')** 则资源根为 **sys._MEIPASS**（轨道 B PyInstaller）；否则资源根为 **os.path.dirname(os.path.abspath(__file__))**（轨道 A Nuitka，__file__ 指向正确执行目录）。  
  - **源码环境**：资源根为 **`__file__`** 所在目录（或根据调用栈/显式传入 package 确定）。  
- **契约**：插件加载自身的 HTML/YAML/TXT/静态资源时，**必须**通过本 API 获取路径，不得在业务中直接写 `sys._MEIPASS` 或仅用 `os.path.dirname(__file__)` 导致打包后或开发时其一失效（见 9.6）。

**标准实现参考**（兼容 PyInstaller、Nuitka 与源码环境）：

```python
# core/path_resolver.py 或 core/config.py

import os
import sys

def get_resource_path(relative_path: str) -> str:
    """兼容 PyInstaller、Nuitka 与源码环境的资源寻址。"""
    if getattr(sys, "frozen", False):
        if hasattr(sys, "_MEIPASS"):
            # 轨道 B：PyInstaller 独有临时解压目录
            base_path = sys._MEIPASS
        else:
            # 轨道 A：Nuitka 编译环境，__file__ 即正确物理路径
            base_path = os.path.dirname(os.path.abspath(__file__))
    else:
        # 源码开发环境
        base_path = os.path.dirname(os.path.abspath(__file__))
    return os.path.join(base_path, relative_path)
```

## 附录：架构设计决策记录（Architecture Decision Appendix）

以下 ADR（Architecture Decision Record）用于记录关键架构决策及其背景，便于后续版本评审与演进。

### ADR-001：桌面宿主技术选型（Tauri + Sidecar）

- **背景**  
  - 需要一个跨平台桌面壳，要求：  
    - 体积小、内存占用低  
    - 能捆绑外部二进制文件  
    - 能与 Web 前端良好集成。
- **备选方案**  
  - Electron（Chromium + Node.js）  
  - 原生 Python GUI（PyQt / Tkinter 等）  
  - 纯 Rust / C++ 自绘 UI
- **决策**  
  - 采用 Tauri V2 + Rust 作为宿主壳，并通过 sidecar 机制启动 Python 网关。
- **影响**  
  - 优点：包体小、资源占用少，与 Web 技术栈契合；Rust 在系统编程上有良好生态。  
  - 风险：对 Tauri 版本演进有依赖，需要关注其长期维护状态。

### ADR-002：网关层技术栈与挂载机制（FastAPI + ASGI Mount）

- **背景**  
  - 需要一个网关中心来聚合自研 API、第三方应用和 MCP Server。
- **备选方案**  
  - 自研 HTTP 网关  
  - Starlette / 纯 Uvicorn + 手动路由  
  - gRPC / 自定义 RPC 框架
- **决策**  
  - 使用 FastAPI 作为网关框架，并通过 `app.mount()` 挂载完整 ASGI 子应用。
- **影响**  
  - 优点：原生支持 OpenAPI，便于前端代码生成；ASGI mount 能零改造承接第三方项目。  
  - 风险：部分非 ASGI 应用需要适配；路由冲突需要在挂载规范里明确约束。  
  - **重要**：Starlette/FastAPI 的 `mount()` 不会将子应用 OpenAPI 合并到主应用，因此必须在网关内实现 **OpenAPI 聚合器**（拦截 `/openapi.json`，拉取主应用 + 各子应用 schema 并合并），否则前端无法通过一次 gen:api 获得全量强类型 SDK（见 4.3）。  
  - 同进程内所有插件共享同一 Python 依赖树；依赖严重冲突的第三方项目建议以 MCP 形态独立进程接入（见 2.3 Non-Goals）。

### ADR-003：前端调用模型（OpenAPI + 代码生成 SDK）

- **背景**  
  - 前端需要在不感知具体插件形态的前提下，统一调用所有能力。
- **备选方案**  
  - 手写 fetch 封装  
  - 使用 GraphQL 网关  
  - 自定义 RPC 协议（如基于 WebSocket 的双向通信）。
- **决策**  
  - 采用 OpenAPI 作为唯一真源，通过 `openapi-ts` 生成 `src-client` SDK，由前端调用。
- **影响**  
  - 优点：类型安全、自动化、高度自描述；新增 / 移除插件后只需重新生成即可。  
  - 风险：OpenAPI 文档体积和聚合复杂度需要控制；对于流式接口需自行约定扩展。

### ADR-004：Python 打包策略（Nuitka vs PyInstaller 双轨）

- **背景**  
  - 既需要小体积 / 高性能 / 反编译防护，又需要兼容 AI / 数据生态中大量动态依赖。
- **决策**  
  - 官方支持两条打包轨道：  
    - Nuitka：推荐用于纯净网关和核心逻辑  
    - PyInstaller：推荐用于复杂插件生态。
- **影响**  
  - 给开发者明确选择空间；在 CI/CD 流程中可根据项目类型选择不同脚本。  
  - 所有插件（A/B 形态）共享同一 Python 环境与依赖；若多个第三方项目依赖版本冲突，本框架不解决「同进程多版本并存」，建议将冲突方改为 MCP 形态独立进程运行（见 2.3）。

### ADR-005：MCP 适配策略（本地 JSON-RPC → HTTP/SSE）

- **背景**  
  - MCP Server 通常基于 stdio + JSON-RPC，不适合直接被 Browser / Next.js 调用。
- **决策**  
  - 在网关层实现 MCP 适配器：  
    - 对前端暴露标准 HTTP 接口 `/api/mcp/...`  
    - 网关内部负责启动 MCP Server、转发 JSON-RPC 请求、聚合结果。
- **影响**  
  - 为前端提供了统一的调用方式和 UI 渲染入口；  
  - 增加了网关实现复杂度，但换来了更好的生态兼容性。  
  - **进程生命周期（悖论四）**：不得仅「提倡」开发者自行实现 Job Object/killpg。框架**必须**在 **core/process_manager.py** 中提供 **ProcessManager**，封装跨平台进程树绑定与 shutdown 时统一回收；MCP 适配器**只能**通过 ProcessManager.spawn() 启动子进程（见 10.4、11.2）。**优雅停机（致命陷阱四）**：Tauri 默认在关窗时直接杀 Sidecar，导致 FastAPI shutdown 永不执行；框架**必须**通过「宿主拦截 CloseRequested → 请求 POST /api/system/shutdown → 再 exit」的握手协议实现优雅停机（见 10.8、4.1、ADR-011）。

### ADR-006：数据持久化路径（Data-Dir 注入）

- **背景**  
  - 单文件打包下，进程在临时目录解压运行，插件若写死 `./data.db` 会导致数据丢失或权限错误；同时全局 **os.chdir(DATA_DIR)** 会破坏 PyInstaller 虚拟文件系统，导致第三方 `open("./config.yaml")` 等读自带资源失败（致命陷阱一，见 10.5）。
- **决策**  
  - 宿主传入 `--data-dir <绝对路径>`；网关**仅**将 **DATA_DIR** 写入 **core/config**，**禁止**在 main.py 级别执行 **os.chdir(DATA_DIR)**。  
  - 数据落盘路径通过各插件在 **setup()** 中适配器/Patch 注入为 `DATA_DIR`；静态资源读取**必须**通过框架提供的 **get_resource_path()**（见 9.6、11.4、11.5），不得硬写 `__file__` 或 `sys._MEIPASS`。
- **影响**  
  - 持久化可预期、跨平台一致；与单文件打包的虚拟文件系统兼容（见 4.5）。

### ADR-007：动态端口分配与 TOCTOU 规避（反向握手）

- **背景**  
  - 若 Rust 先 `bind("127.0.0.1:0")` 取得端口再释放并传给 Python，在释放到 Python bind 之间的「真空期」内端口可能被系统分配给其他进程，导致 Python 启动失败（TOCTOU）。
- **决策**  
  - 采用**反向握手**：宿主传 `--port 0` 给网关，由 Uvicorn 在底层自行绑定空闲端口；因 uvicorn.run 为阻塞调用，宿主**从 Uvicorn 原生日志**（`Uvicorn running on http://127.0.0.1:<port>`）解析端口并保存，供前端查询（见 9.1）。
- **影响**  
  - 占端口与使用端口由同一进程在同一时刻完成，无竞态；实现见 4.1、9.1。  
  - **深坑 9.5**：宿主启动 sidecar 时**必须**注入 **PYTHONUNBUFFERED=1**，否则 Python 子进程 stdout 缓冲会导致 Uvicorn 日志迟迟不 flush、端口解析死等（见 4.1、9.5）。

### ADR-008：本地 CORS 与可选 Token 鉴权

- **背景**  
  - 网关若使用 `allow_origins=["*"]`，恶意网页可通过浏览器向 `http://127.0.0.1:<port>` 发起跨域请求，冒充本地应用调用 MCP/插件 API（本地 CSRF/SSRF）。
- **决策**  
  - **必选**：CORS 的 `allow_origins` 严格限制为 Tauri 专属来源（如 `tauri://localhost`、`http://tauri.localhost`），禁止 `["*"]`。  
  - **强烈建议**：宿主生成随机 Token，经 `--auth-token` 传入网关；网关通过**鉴权中间件**校验 Token。  
  - **悖论三**：浏览器 WebSocket API 无法设置 Header，故鉴权中间件**必须协议分流**：HTTP 用 `Authorization: Bearer`；WebSocket 必须支持从 **URL query（?token=xxx）** 或 **Subprotocols** 提取 Token。框架在 **core/middlewares/auth.py** 中提供现成实现（见 10.3、11.3）。
- **影响**  
  - 仅 Tauri WebView 可跨域调用；WebSocket 类第三方插件（实时终端、流式对话）不被误拦（见 6.2、9.2）。

### ADR-009：插件化与动态加载契约

- **背景**  
  - 若每次新增插件都在 main.py 中手写 import 与 mount，仅为「目录隔离」而非真正插件化，维护成本高且易出错。
- **决策**  
  - 网关主干（main.py）**不因新增/删除插件而改动**。采用 **importlib** 等对 `plugins/` 目录做动态扫描；每个插件通过**契约**暴露挂载入口（如 `manifest.json` 或统一函数 `setup(app: FastAPI)`），网关启动时遍历并调用，完成路由/Mount 挂载。  
  - **悖论一**：Nuitka/PyInstaller 无法追踪 importlib 动态引用，打包会漏掉插件代码。**必选**：在构建前运行 **scripts/pre_build_scanner.py**，生成显式 import 或 --hidden-import，使静态打包与动态加载兼容（见 10.1、11.1）。  
  - **致命陷阱三**：扫描器**必须**同时生成 **--add-data**（PyInstaller）/ **--include-data-dir**（Nuitka）等静态资源参数并注入构建脚本，否则插件内 templates/、*.yaml 等会丢失（见 10.7、11.1）。
- **影响**  
  - 新增插件 = 在 plugins 下新增符合契约的目录；**打包前必须执行预编译扫描**（含静态资源），否则产物缺插件代码或静态资源（见 9.3）。

### ADR-010：OpenAPI 聚合与 Schema 命名空间重写

- **背景**  
  - 多插件合并 OpenAPI 时，若直接深拷贝合并 `components/schemas`，不同插件的同名模型（如 `User`）会相互覆盖，导致生成的前端类型与真实接口错位。
- **决策**  
  - 聚合器在合并**每个**子应用 schema 时，必须做 **命名空间重写**：为该子应用分配前缀（如挂载路径规范化），将所有 schema 名称改为 `<Prefix>_<原名>`，并递归重写该子应用 paths 内所有 `$ref` 指向，再并入全局 components。
- **影响**  
  - 多插件强类型 SDK 可正确生成，无命名冲突（见 4.3、9.4）。

### ADR-011：优雅停机握手协议（Graceful Shutdown）

- **背景**  
  - 用户点击 Tauri 窗口「X」时，Tauri 默认会直接杀死 Sidecar 进程（Windows 下为 TerminateProcess），导致 **FastAPI lifespan/shutdown 钩子均不执行**，ProcessManager.terminate_all() 永远不会被调用（致命陷阱四，见 10.8）。Windows 无控制台下 **signal.raise_signal(SIGINT)** 易触发强杀（终极死角二）；stdin 管道看门狗在 Tauri 下易导致启动即 EOF「光速自杀」（终极死角一）。  
- **决策**  
  - **网关**：**编程式**创建 **uvicorn.Server**，将实例注入 **app.state.server**；在 **system_api/router.py** 中暴露 **POST /api/system/shutdown**，收到后设置 **request.app.state.server.should_exit = True**，**禁止**依赖 signal 或 sys.exit()。宿主猝死场景下，main.py 以 **--host-pid** 轮询宿主存活，发现宿主已死时同样设置 **server.should_exit = True**（见 11.2 附、11.2 附二）。  
  - **宿主**：启动 Sidecar 时传入 **--host-pid \<当前 PID\>**；在 **main.rs** 中拦截 **WindowEvent::CloseRequested** → 请求 **POST /api/system/shutdown** → 等待响应并再挂起约 1.5 秒 → **app.exit(0)**（见 4.1、10.8）。  
- **影响**  
  - 关闭窗口或宿主猝死时，网关均通过 **should_exit** 走完 lifespan shutdown，ProcessManager.terminate_all() 与资源释放可预期；实现见 4.1、10.8、10.9、11.2、11.2 附、11.2 附二。
