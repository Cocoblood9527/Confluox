# 实例说明

## 这份文档解决什么问题

这份文档给出 3 个可直接参考的接入案例，帮助你快速判断：

- 该用 `api in_process`、`api out_of_process`，还是 `worker`
- 最小 `manifest.json` 应该怎么写
- 前端应当通过什么路径调用

所有示例都以当前仓库已经支持的能力为准。

## 术语对齐

- `api in_process`：API 插件在网关进程内运行
- `api out_of_process`：API 插件由网关拉起独立进程并代理请求
- `worker`：受管后台进程，不自动暴露 HTTP 路由
- `permissions`：worker 启动前的权限声明，按 host allowlist 校验
- `sandbox_profile`：worker 轻量沙箱硬化档位（如 `restricted`、`strict`）
- `X-Confluox-Plugin-Auth`：host 到 out-of-process API 插件的内部鉴权头

## 案例 1：Whisper 本地转录（`api out_of_process`）

适用场景：

- 模型依赖重、内存占用高
- 希望插件进程和主网关隔离
- 需要稳定的 HTTP 接口给前端调用

推荐插件清单：

```json
{
  "type": "api",
  "name": "whisper_app",
  "entry": "entry:setup",
  "execution_mode": "out_of_process",
  "command": ["python3", "-m", "whisper_app.server"]
}
```

后端契约（最小示意）：

```python
import os

from fastapi import APIRouter, Depends, FastAPI, Header, HTTPException

app = FastAPI()
plugin_prefix = os.environ.get("CONFLUOX_PLUGIN_PREFIX", "/api/whisper_app")
plugin_token = os.environ.get("CONFLUOX_PLUGIN_AUTH_TOKEN", "")


def require_host_auth(
    header_token: str | None = Header(default=None, alias="X-Confluox-Plugin-Auth"),
) -> None:
    if header_token != plugin_token:
        raise HTTPException(status_code=401, detail="invalid host auth")


@app.get("/__confluox/health", dependencies=[Depends(require_host_auth)])
def health() -> dict[str, str]:
    return {"status": "ok"}


router = APIRouter(prefix=plugin_prefix, dependencies=[Depends(require_host_auth)])


@router.post("/transcribe")
def transcribe(payload: dict[str, str]) -> dict[str, str]:
    # 示例：实际项目里在这里调用 whisper 推理
    return {"text": f"transcribed: {payload['audio_path']}"}


app.include_router(router)
```

前端调用（当前客户端契约）：

```ts
const result = await apiPost<{ text: string }>('/api/whisper_app/transcribe', {
  audio_path: selectedPath,
  language: 'zh',
})
```

注意：

- 当前 `apiPost` 是 JSON 通道，不是 `FormData` 上传通道。
- 如果你要做文件上传，请单独设计上传接口和客户端处理方式。

## 案例 2：后台索引任务（`worker`）

适用场景：

- 长时间后台任务
- 不需要把 worker 本身暴露为 HTTP 服务
- 需要权限声明和受控进程生命周期
- 场景以内网/本地处理为主

推荐 `worker` 清单：

```json
{
  "type": "worker",
  "name": "index_worker",
  "runtime": "python",
  "permissions": {
    "fs": ["read:/tmp"],
    "network": ["loopback"]
  },
  "sandbox_profile": "restricted",
  "command": ["python3", "-m", "index_worker.main"]
}
```

实现建议：

- `worker` 负责后台扫描、索引或队列消费任务。
- 再配一个轻量 `api` 插件读取状态并暴露给前端。
- 前端只调用 API 插件，不直接感知底层子进程细节。

## 案例 3：Markdown 生成器（`api in_process`）

适用场景：

- 路由简单
- 依赖较轻
- 更看重接入成本和开发效率

推荐插件清单：

```json
{
  "type": "api",
  "name": "md_builder",
  "entry": "entry:setup"
}
```

后端示意：

```python
from fastapi import APIRouter


def setup(context) -> None:
    router = APIRouter(prefix="/api/md")

    @router.post("/build")
    def build(payload: dict[str, str]) -> dict[str, str]:
        # 示例：实际项目里调用本地 markdown 构建流程
        output_path = f"{context.data_dir}/site/index.html"
        return {"status": "ok", "output": output_path}

    context.app.include_router(router)
```

前端调用：

```ts
await apiPost('/api/md/build', { source_dir: '/path/to/docs' })
```
