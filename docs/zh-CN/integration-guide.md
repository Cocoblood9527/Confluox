# 开源项目接入指南

## 先建立正确预期

Confluox 是一个本地能力桥接框架，不是“任何开源项目零改造桌面化”的承诺。

真正合适的接入方案，取决于你要带进来的项目属于哪一类。

## 路径 1：轻量 API 项目

最适合：

- FastAPI 应用
- 基于 APIRouter 的工具
- 边界清晰的小型 Python 服务

推荐做法：

- 提取或包装真正有用的路由能力
- 通过插件的 `setup(context)` 暴露出来
- 状态数据写入框架提供的数据目录
- 前端通过网关插件接口调用

这是当前成本最低、最成熟的路径。

## 路径 2：CLI 工具或本地服务

最适合：

- 命令行工具
- 任务执行器
- 本地推理或自动化进程
- 需要保持独立进程形态的工具

推荐做法：

- 在外面包一层 adapter
- 通过框架控制的生命周期来启动和管理进程
- 在网关层暴露一小层对前端友好的 API
- 不要把原始子进程细节直接泄漏给 UI

这类接入通常是中等成本。

## 路径 3：大型第三方系统

最适合：

- 自带前端的项目
- 依赖树很重的系统
- 默认假设自己是主运行时的应用
- 带复杂流式协议或会话模型的服务

推荐做法：

- 用 adapter 或 proxy 先把边界隔离出来
- 尽量不要直接并入主网关进程
- 按能力边界逐步集成
- 不要过早追求源码级彻底统一

这是成本最高的一条路径，需要有意识地规划。

## 一个实用判断法

先问自己几个问题：

- 它能不能已经像一个本地 API 一样工作？
- 它能不能安全地以受控子进程方式运行？
- 它是不是强依赖自己的运行时假设？
- 我们是否只需要接它的一部分能力，而不是整个系统？

如果第一个问题的答案是“能”，就优先从 API 插件开始。

## 需要避免的做法

- 对大型项目预期零改造
- 直接把完整第三方系统 import 到主网关进程
- 依赖全局工作目录
- 跳过数据目录和资源路径注入
- 在接入边界还没稳定前就先做复杂 UI
